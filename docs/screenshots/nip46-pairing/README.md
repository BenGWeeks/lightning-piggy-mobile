# NIP-46 pairing screenshots

Visual verification of the NIP-46 ("Nostr Connect") signer (issue #283), captured on
**both platforms** — the Android set below is Maestro + adb; the iOS set (`ios-*.png`) is
captured on the iOS Simulator, the primary NIP-46 target (Clave is iOS-only; Amber is
Android-only).

## Android (`com.lightningpiggy.app.dev`, Maestro-driven)

- `01-login-sheet-with-nip46-button.png` — the Connect Nostr sheet showing the new
  "Use NIP-46 Signer (Aegis / Nowser / nsec.app)" button below the nsec input + Amber button.
- `02-pairing-qr-screen.png` — the NIP-46 pairing screen with the `nostrconnect://` QR
  rendered + the "Waiting for bunker…" spinner (waits up to 120s for the bunker's `connect` ack).
- `03-pairing-timeout.png` — the sheet after the 120s window elapses without an ack,
  surfacing the user-facing "Pairing took too long — try again" error.

## iOS (`ios-*.png`, iPhone 17 Pro simulator, iOS 26.4)

Captured on a locally-built iOS dev build. These show the **iOS-specific UI** the Android
shots can't: the button reads "Use NIP-46 Signer (**Clave** / Aegis / nsec.app)" — Clave
first, the iOS-native signer — with **no Amber button** (Amber is Android-only), and every
string is localized (the Android shots predate the i18n pass and render the button as
untranslated English inside an otherwise-Spanish UI).

- `ios-01-login-sheet.png` — the Connect Nostr sheet on iOS with the Clave-first NIP-46 button.
- `ios-02-pairing-qr.png` — the "Connect via NIP-46" QR screen ("Open your bunker app
  (Clave, Aegis, nsec.app) and scan this QR…"), `nostrconnect://` QR + "Waiting for bunker…",
  with the **"Open in bunker app"** button (same-device deep-link hand-off) above Copy/Cancel.
- `ios-03-pairing-timeout.png` — the localized "Pairing took too long — try again" error.

### iOS test outcome

The app runs on the iOS emulator and the pairing flow works through the UI. The
`nostrconnect://` URI generated by the running iOS build was extracted and is spec-correct:
`nostrconnect://<client-pub>?relay=wss://relay.nsec.app&secret=<32hex>&perms=sign_event,nip04_encrypt,nip04_decrypt,nip44_encrypt,nip44_decrypt&name=Lightning+Piggy`.
**Clave** (built from source, `DocNR/clave`, running on the same simulator) parses that exact
URI shape and reaches its "Approve Connection" sheet. Full pairing completion couldn't be
finished in-simulator: Clave's signing path is a push-woken Notification Service Extension
that needs production APNs (unavailable on a simulator — per Clave's own README, only the
handshake works in debug), and the login sheet's `@gorhom/bottom-sheet` content is not in
the iOS accessibility tree, so Maestro can't drive it (**follow-up: expose the sheet content
so the iOS NIP-46 e2e is coordinate-free**). The signer service itself is covered by the
`nostrConnectService` / `useNip46Login` unit tests (pairing-URI shape, sign / nip04 / nip44
encrypt+decrypt, the adopt/abort lifecycle, and the restore shape-guard).

## Test outcome (nsec.app cross-check)

The **app side works correctly end-to-end**: it generates a well-formed
`nostrconnect://` URI (client pubkey, `relay=wss://relay.nsec.app`, secret, the five
`sign_event` / `nip04_*` / `nip44_*` perms, `name=Lightning Piggy`), renders it as a QR,
and subscribes for the bunker's ack.

When that URI is pasted into nsec.app's **Connect app** dialog, nsec.app **correctly
recognises it** and shows a "Connection request — Lightning Piggy — New app would like to
connect — Asking 5 permissions" prompt, confirming the URI format + perms are valid and
interoperate with a real bunker.

Full pairing did **not** complete in the sandboxed test environment: after approving in
nsec.app, the bunker's `connect` ack never arrived back at the app over
`wss://relay.nsec.app`, so the app timed out (shot 03). This traced to constrained
connectivity to nsec.app infrastructure from the sandbox (nsec.app's own name/nip05
backend returned `Failed to fetch`), not a defect in the app — the app's timeout + error
handling behaved as designed. A separate machine is validating the iOS Clave signer path.
