== Nostr Protocol State

This page documents which Nostr Improvement Proposals (NIPs) Lightning Piggy Mobile implements, in which direction (send / receive), and any deprecation timelines.

=== DM Protocol State

==== Summary

Lightning Piggy ships **NIP-17 gift-wrapped DMs** (kind-1059 → 13 → 14) as the **only** outbound 1:1 messaging path. Legacy **NIP-04** (kind 4) events are **read** for backward-compatibility with peers on older clients but are **never sent**. Group chats also use NIP-17 (a multi-`p`-tag kind-14 rumor).

[cols="1,1,1,3", options="header"]
|===
| Direction | NIP-04 (kind 4) | NIP-17 (kind 1059 → 13 → 14) | Notes

| Send (1:1 DM) | ❌ removed | ✅ only path | `sendDirectMessage` in `src/contexts/NostrContext.tsx`
| Send (group) | n/a — never supported | ✅ only path | `sendGroupMessage` in `src/contexts/NostrContext.tsx`
| Receive (display) | ✅ kept | ✅ kept | `fetchConversation`, `refreshDmInbox` query both kinds and merge into one thread sorted by the rumor's `created_at`
|===

==== Why NIP-17

* NIP-04 is marked `unrecommended` at the top of the NIP itself. AES-256-CBC, no metadata protection — sender pubkey, `created_at`, recipient `p` tag are all on the wire in plaintext.
* NIP-17 wraps each kind-14 rumor in a kind-13 seal inside a kind-1059 gift wrap. The outer wrap exposes only a random ephemeral pubkey, a `created_at` randomised up to 2 days in the past, the `kind: 1059` tag, and the recipient `p` tag. Sender identity, message kind, true send time, and any tags on the rumor are all hidden from relays and observers.
* Encryption throughout NIP-17 is **NIP-44** v2 (secp256k1 ECDH → HKDF → ChaCha20 + HMAC-SHA256).

==== Send path (outgoing)

* Inner rumor is built by `createDirectMessageRumor` (1:1) or `createGroupChatRumor` (groups) — both produce a `kind: 14` event with a `pubkey` field but no signature.
* `sendNip17ToManyWithNsec` (nsec signer) seals + wraps the rumor for each recipient via `nostr-tools` `nip59.wrapManyEvents`, which also wraps for the sender so the message lands in the user's own inbox on other devices.
* `sendNip17ToManyWithSigner` (Amber signer) does the same SEQUENTIALLY because Amber's native module rejects concurrent intents with `BUSY`. Per recipient: 1× `nip44Encrypt` + 1× `signEvent` for the seal; the gift wrap is signed locally with a fresh ephemeral key (never the user's key) so it does not need a signer round-trip.
* Wraps are published in parallel via `pool.publish` to the union of the user's NIP-65 write relays and `DEFAULT_RELAYS`.

==== Receive path (display)

The Messages tab fetches both protocols on focus and on conversation open:

* `fetchInboxDmEvents` queries three filters in parallel:
+
[source]
----
{ kinds: [4],    authors: [me] }   // outgoing legacy DMs
{ kinds: [4],    '#p':     [me] }   // incoming legacy DMs
{ kinds: [1059], '#p':     [me] }   // gift wraps addressed to us
----
* Kind-4 events are NIP-04-decrypted with the appropriate signer (`nip04.decrypt` for nsec, Amber's `nip04Decrypt` IPC for Amber) and merged into the per-peer thread.
* Kind-1059 wraps are unwrapped via `unwrapWrapNsec` (nsec) or `unwrapWrapViaNip44` (Amber) — both perform the full NIP-17 verification chain: signature check on the wrap, NIP-44-decrypt the wrap to a kind-13 seal, NIP-44-decrypt the seal to a kind-14 rumor, and assert `rumor.pubkey === seal.pubkey` (the spec MUST that prevents sender-spoofing on a tampered rumor).
* `classifyRumor` then routes 1:1 rumors into the per-peer DM thread and multi-recipient rumors into the matching group room.
* Threads sort by the **rumor's** `created_at`, never the gift wrap's (the wrap timestamp is intentionally fake per the spec).

==== Deprecation timeline for NIP-04 reads

We will keep the kind-4 read filter and the `nip04.decrypt` codepath until both of:

. The user research signal that no active contacts are still on NIP-04-only clients (Damus is the relevant laggard at the time of writing).
. The kind-4 plaintext cache (`nip04PlaintextCache` LRU + the per-thread persisted cache) for a representative active user shows zero new fresh-decrypt entries for at least 90 days, evidenced by the `[Perf] fetchConversation` log line `k4=N (hits=0, fresh=0)`.

There is no scheduled removal date. Removal will be announced one release ahead in the GitHub Release notes and the read filter will land behind a feature flag for one release before the codepath itself is deleted.

==== Interop notes

* **Damus** (as of April 2026 master) does not subscribe to kind 1059 — its `HomeModel` filter is kind-4-only. A message Lightning Piggy sends to a Damus-only peer will not appear in their UI until Damus ships NIP-17. Other major clients (Amethyst, 0xchat, Snort, Coracle, Primal) accept NIP-17 and will display Lightning Piggy's outgoing DMs.
* The `subject` tag on a kind-14 is what 0xchat / Amethyst / Quartz read to display the conversation title — `createGroupChatRumor` always emits one; `createDirectMessageRumor` deliberately omits it (a 1:1 thread has no name). See issue #271 for the cross-client subject-tag interop work.
* Lightning Piggy does **not** publish a kind-10050 NIP-17 DM relay-list event. We rely on the user's existing NIP-65 write-relay list plus the DEFAULT_RELAYS fallback. If a recipient publishes their own kind-10050 we do not currently honour it — they must accept wraps on at least one relay we publish to.

=== Hunt: Lightning extension to NIP-GC (#468)

==== Summary

Lightning Piggy's "Hunt" feature lets users hide LNURL-withdraw "Piggies" in
the wild and lets others find + claim them. The wallet-level LNURL-w is
created by the hider in their own wallet (LNbits / Alby / Mutiny / …) — see
the project memory `No LNbits-specific APIs`. Lightning Piggy is the
**publisher**, **discoverer**, and **find-log thread** layer.

We adopt **NIP-GC** (Nostr Geocaching, draft by https://treasures.to[treasures.to])
verbatim and mark Lightning-payout caches with a **NIP-32 label**
(`L=com.lightningpiggy.app` / `l=payout-lnurl-w`). The LNURL bearer
token is deliberately **never** put on the public event — it lives
only on the physical NFC tag / QR and the hider's local SecureStore;
publishing it on a kind 37516 would let any relay subscriber drain
the cache without ever physically visiting it (see
`feedback_lnurl_never_on_relays.md` and the
`buildCacheListing` security test in `nostrPlacesService.test.ts`).
A Hunt Piggy is therefore just a NIP-GC `traditional` cache flagged
by our label; generic geocaching clients (treasures.to et al.)
ignore the label and render it as a normal cache, while LP routes
the finder through HuntFoundScreen for claim. The `wait` / `uses`
fields exist on the local `HiddenPiggy` record for the cooldown UX
but are NOT published as event tags — that was an early design
draft that got dropped to avoid side-channel discoverability of
the cache's economics. **Read the
upstream draft** in the kind-30817 event with `d="geocaching-events"`
authored by `npub1pse...zz68` (or
https://nostrhub.io/naddr1qvzqqqrcvypzppscgyy746fhmrt0nq955z6xmf80pkvrat0yq0hpknqtd00z8z68qqgkwet0vdskx6rfdenj6etkv4h8guc6gs5y5[the
nostrhub.io rendering]).

The data model uses **three Nostr kinds with three different lifecycles**:

[cols="1,1,1,3", options="header"]
|===
| What | Kind | Lifecycle | NIP source

| Cache listing | `37516` | **Replaceable** (PRE — `30000-39999` range, keyed by `(pubkey, kind, d)`). Hider can re-publish to update LNURL, location, hint, cooldown, expiry, etc.; relays drop the prior version. | NIP-GC §"Geocache Listing Event"
| Found-log entry | `7516` | **Append-only**, `7000-7999` regular range. Each find produces one event tagged to the cache's `a` coordinate. Optional embedded `kind 7517` verification event for cryptographic find-proof. | NIP-GC §"Found Log Event"
| Comment / status (DNF, note, maintenance, archived) | `1111` | **Append-only**, NIP-22 comment threading model with the cache as both root + parent. | NIP-GC §"Comment Log Events" + NIP-22
|===

==== kind 37516 listing — full tag schema

NIP-GC required tags (we always emit):

[cols="1,3", options="header"]
|===
| Tag | Meaning

| `["d", piggy_id]` | Stable identifier scoped to the hider's pubkey. Format: `piggy_<base36-millis>_<random6>` (see `newPiggyId` in `src/services/piggyStorageService.ts`).
| `["name", str]` | Short human-readable cache name. Required by NIP-GC.
| `["g", geohash]` (×7) | Location at every precision from 3 to 9 chars — cheap on event size, dramatically widens prefix-filter surface for proximity queries. See `src/utils/geohash.ts`.
| `["D", "1-5"]` | Finding difficulty. We default `1` (just-find-the-tag) for NFC-stash Piggies; user can override for tricky hides.
| `["T", "1-5"]` | Physical terrain. Default `1` (urban/accessible); bump for hilltop / hike-required caches.
| `["S", "micro\|small\|regular\|large\|other"]` | Cache size. Default `micro` since an NFC sticker is small.
|===

NIP-GC optional tags (we emit when present):

[cols="1,3", options="header"]
|===
| Tag | Meaning

| `["t", "traditional\|multi\|mystery"]` | Cache type. Default `traditional` per NIP-GC. **All Lightning Piggies are traditional caches with a Lightning bonus.**
| `["hint", rot13(str)]` | Plaintext clue, ROT13-encoded per NIP-GC client guidance (prevents inline spoilers in scrolling clients). LP decodes for display; users with non-NIP-GC-aware clients see the rotated text and need to ROT13 themselves.
| `["image", url]` | EXIF-stripped Blossom / nostr.build URL of the hider's hint photo. Routed through `src/services/imageUploadService.ts`.
| `["expiration", unix_ts]` | NIP-40. Clients hide the listing past this time without needing a separate delete event.
|===

LP marker tags (NIP-32 label, **not** a bearer token — the LNURL deliberately does NOT go on the public event):

[cols="1,3", options="header"]
|===
| Tag | Meaning

| `["L", "com.lightningpiggy.app"]` | NIP-32 label namespace declaring "this cache is in the Lightning Piggy taxonomy".
| `["l", "payout-lnurl-w", "com.lightningpiggy.app"]` | NIP-32 label value within the namespace — declares the cache pays out via LNURL-withdraw on the physical artifact (the NFC tag / printed QR carries the actual LNURL string). Generic NIP-GC clients (treasures.to) ignore the label entirely and render the cache as a standard listing; LP recognises it and renders the 🐷 pin + "tap the physical tag to claim" UX.
|===

**The LNURL never goes in the public event** — that would broadcast a bearer token to every relay subscriber, letting anyone in the world drain the cache without ever physically visiting it (defeating the entire physical-cache premise). The LNURL lives on:

* The **physical artifact** (NFC tag / printed QR) — this is the real access control. Possession = consent to claim.
* The **hider's local `SecureStore`** (`piggyStorageService.HiddenPiggy.lnurlw`) — for tag-write, re-print, and any future "show me the QR for this Piggy" affordance. Never leaves the device.

This is how TapTheSatsMap (the most prolific NIP-GC publisher, 31 sticker caches) operates — see project memory `treasures.to interop` for the analysis. Wait_time / uses / per-claim caps remain the hider's responsibility to set in their own wallet's LNURL-w configuration; LP doesn't try to mirror them on the event because doing so contributes to side-channel discoverability of the cache's economics without adding any safety.

`content` carries the longer cache description ("🐷 Geo-Cache 1 — 21 sats per claim, 3h cooldown, 100 uses.").

==== kind 7516 found-log — what we emit

Each successful claim produces:

* `["a", "37516:<hider_pubkey>:<piggy_id>"]` — required NIP-GC reference to the cache.
* `["image", url]` (optional) — finder's photo of the find spot. Same EXIF-strip pipeline as the listing's hint photo.
* `["amount", "21"]` (LP extension) — sats claimed via the LNURL-w. Drives the `⚡ claimed` badge and "you got 21 sats here 47m ago" copy on the detail screen. Treasures.to et al. ignore this; for them it's a regular found-log.
* `["verification", <kind-7517-json>]` (optional) — embedded NIP-GC verification event signed by the cache's verification key, if the listing carries one. **For LP Piggies we don't need this** — successful claim of the LNURL-w is its own physical-presence proof (you had the bearer token, the issuer paid you).

==== Subscribe filter shape

* **Discovery feed** — `{ kinds: [37516], "#g": [<geohash prefixes for current viewport>] }`. Single round-trip catches both LP Piggies AND treasures.to caches in the same area (we filter for the `["l", "payout-lnurl-w", "com.lightningpiggy.app"]` label client-side to render the 🐷 vs 📍 pin distinction). Prefix list comes from `geohashPrefixes(currentGh, 5)` so one subscribe covers a ~5 km city neighbourhood.
* **Single cache detail (replaceable + thread)** — three filters:
  ** `{ kinds: [37516], authors: [hider], "#d": [piggy_id] }` — latest revision of the listing.
  ** `{ kinds: [7516], "#a": ["37516:<hider>:<piggy_id>"] }` — append-only found-log thread.
  ** `{ kinds: [1111], "#A": ["37516:<hider>:<piggy_id>"] }` — append-only comment thread (DNF / notes / maintenance / archived).
* **My Piggies (cross-device)** — `{ kinds: [37516], authors: [me] }` and filter the result for the NIP-32 `["l", "payout-lnurl-w", "com.lightningpiggy.app"]` label marker. **Not** the `lnurl` tag — that one deliberately never reaches the relay (see the "marker tags" subsection above).

==== Why NIP-GC rather than a custom kind

We initially picked a custom `kind 30408` (still in the PRE range, semantically equivalent). Switched to NIP-GC `kind 37516` after a one-line user direction:

> _"i think we should use their NIPs and use traditional but with an LNUrlw additional field."_

Reasons it's the right call:

* **Network effects.** treasures.to users see our Piggies on their map, vice versa. One geocaching ecosystem on Nostr instead of two competing ones.
* **No semantic conflict.** A Lightning Piggy *is* a traditional NFC-tag cache — it just also pays. The `lnurl` extension is purely additive; their schema doesn't need to change.
* **Their schema is more thought-through.** D/T/S/t classification, ROT13 hint convention, NIP-22 comment-thread model, optional cryptographic verification (kind 7517) all generalise beyond what we'd built. We inherit it for free.
* **Migration cost: trivial.** `30408` → `37516` is a single-kind change; the tag schema is a strict superset of what we had; one orphan test event from the old kind hits its 30-day NIP-40 expiration and ages out.
