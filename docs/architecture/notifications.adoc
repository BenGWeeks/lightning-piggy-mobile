= OS Notifications Architecture
:toc:
:source-highlighter: rouge

This document explains how Lightning Piggy delivers OS-level
notifications (incoming DMs, group messages, payments, zaps) to a
backgrounded or closed app *without* depending on Firebase Cloud
Messaging or any other Google Play Services component.

It is the canonical reference for the foundation landed in PR for
issue https://github.com/BenGWeeks/lightning-piggy-mobile/issues/279[#279].

== Why no FCM

Lightning Piggy targets users on de-Googled Android distributions
(GrapheneOS, CalyxOS, /e/OS, microG-flashed stock ROMs). FCM
requires Google Play Services to be present on the device — when GMS
is absent, an FCM-based notification stack silently never fires, and
the affected user has no way to fix it short of reflashing.

Standard React Native push libraries (`react-native-firebase/messaging`,
`@notifee/react-native` with FCM transport, OneSignal, etc.) all sit
on top of FCM and inherit this constraint. They are excluded from
Lightning Piggy's dependency graph by policy, not just preference.

Apple Push Notification service (APNs) is the equivalent on iOS but
requires us to operate a remote push server that holds an APNs token
per device and forwards events to it. We deliberately do not run a
backend (the app talks to relays + LNbits + Boltz directly), so APNs
is out for the same architectural reason that FCM is out: there is
no remote sender. iOS notifications therefore arrive via local
notifications fired by the JS runtime when it next runs (see
"iOS background story" below).

== The three pieces

[cols="1,3"]
|===
| Component | Role

| `src/services/notificationService.ts`
| Single integration point. Exposes typed `fireNotification(payload)`
to every trigger site (DM rumor decrypt, payment settle, zap receipt).
Owns Android channels, the foreground permission flow, and the
lock-screen content privacy preference. Backed by `expo-notifications`
LOCAL notifications (no FCM token requested).

| `expo-background-task` + `src/services/backgroundSyncService.ts`
| The cross-platform background worker — WorkManager on Android,
BGTaskScheduler on iOS — registered in `src/services/backgroundTask.ts`.
When the UI isn't mounted the OS periodically runs `runBackgroundSync`,
which does DETECT-AND-PING: it notices new encrypted inbound traffic
and fires a generic "new message" notification *without decrypting*.
No custom native code; the config plugin wires the platform bits.

| Android foreground service (future upgrade)
| A persistent relay WebSocket for *realtime* (vs ~15-min) delivery.
Manifest permissions are scaffolded in
`plugins/withForegroundService.js`, but the Kotlin `Service` class is
not implemented — the `expo-background-task` path above is what ships.
|===

== Android background story

----
[ WorkManager ] ──(~15 min)──> runBackgroundSync()
                                   │
                                   ├── queries relays for new kind-1059
                                   │   gift wraps + kind-4 DMs addressed
                                   │   to us (DOES NOT decrypt)
                                   │
                                   └─> notificationService.fireMessageNotification({...})
                                          │  (generic "New message" copy)
                                          └─> expo-notifications local schedule
                                                 │
                                                 └─> OS notification shade
----

* The background worker is scheduled by `expo-background-task`
  (WorkManager under the hood). ~15-minute floor — Android won't run a
  periodic task more often. That cadence suits detect-and-ping: we only
  need to notice new traffic arrived, not stream it.
* DETECT-AND-PING, never decrypt in the background. This works for BOTH
  nsec and Amber users (Amber can't decrypt headlessly), and keeps
  plaintext off-screen. See `backgroundSyncService.ts`.
* `POST_NOTIFICATIONS` runtime permission is requested from the
  FOREGROUND at app launch — not lazily from a fire path, since a
  headless task can't present the OS dialog. See
  `notificationService.requestNotificationPermission()`; fire paths
  only ever call `hasNotificationPermission()`.
* A persistent foreground service (realtime relay socket) is a possible
  future upgrade — `FOREGROUND_SERVICE` / `FOREGROUND_SERVICE_DATA_SYNC`
  / `RECEIVE_BOOT_COMPLETED` permissions are scaffolded in
  `withForegroundService.js`, but the Kotlin `Service` is not built.

This path works on stock Android, GrapheneOS, microG, and any other
AOSP derivative. It does not work in scenarios where the user has
manually battery-restricted the app — that's a documentation
problem, addressed in onboarding.

== iOS background story

iOS has no equivalent to Android's foreground service. The two
alternatives we evaluated:

1. **Silent push via APNs** — requires us to run a remote server
   that holds APNs tokens and forwards events. *Rejected* — same
   architectural reason as FCM (no backend).
2. **`audio` background mode keep-alive trick** — plays a near-silent
   audio stream so iOS keeps the app process alive. App Store
   review reliably rejects this. *Rejected.*
3. **`BGTaskScheduler` periodic refresh** — OS-controlled cadence,
   typically 15-30 min between executions, may be longer if the
   device is locked / on battery / hasn't been used recently.
   *Accepted as the iOS reality.*

Trade-off the user must understand: on iOS, message and payment
notifications can be delayed by up to ~30 min, and the device must
have been used recently for the OS to schedule the task at all.
This will be explained in iOS onboarding when the iOS build ships.

== Privacy: lock-screen content

Decrypted DM plaintext is *never* written to the notification body by
default. The user must explicitly opt in via the Settings toggle on
SecurityScreen (preference key `notif_pref_lockscreen_content_v1`).

How it's enforced:

* **Per-notification level (the real guard)** — when the preference is
  OFF (default), `notificationService.fireNotification` substitutes the
  caller-provided title/body with a generic
  `"New message" / "Open Lightning Piggy to read"` pair before
  scheduling. The actual content never enters the OS notification store
  at all, even briefly. This is what makes the opt-in toggle meaningful.
* **Android channel level** — both `messages` and `payments` channels
  use `lockscreenVisibility: AndroidNotificationVisibility.PRIVATE`
  (not `SECRET`). PRIVATE lets the notification appear on the lock
  screen while honouring the user's system-level "hide sensitive
  content" setting. We deliberately avoid `SECRET`: it would hide the
  entry entirely even after the user opts into lock-screen content,
  making the toggle a no-op on Android.

Only routing metadata rides in the notification's `data` field — the
`kind` plus conversation/group/wallet ids — *not* the plaintext title
or body. The in-app screen we navigate to on tap reloads the real
content from the local store once the user has unlocked.

== Per-channel mute

Android 8+ surfaces every channel separately in system Settings →
Apps → Lightning Piggy → Notifications. A user who wants payment
alerts but no message alerts (or vice versa) can mute either channel
without affecting the other or uninstalling the app.

In-app per-channel toggles are a follow-up — for the foundation PR,
the system Settings UI is the only mute path.

== The notification-manager service as single integration point

Every trigger site in the app — `NostrContext.tryRouteGroupRumor`,
the future DM-inbox notification fire, payment-settled handlers,
zap-received handlers — calls `fireNotification(payload)` and
nothing else. They never reach into `expo-notifications` directly.

This seam exists so we can:

* swap the backing notification library without touching trigger
  sites,
* add notification grouping / collapsing in one place (multiple
  DMs from the same sender → one notification),
* implement quiet hours / per-contact mute / snooze in one place,
* add badge counts in one place,

— all without an audit pass through every trigger site.

== Status (as of the foundation PR)

Landed:

* `expo-notifications` dependency wired (SDK 55-compatible version)
* `notificationService` with Android channels, foreground permission
  flow, privacy-preserving lock-screen behaviour, typed payload API
* All triggers wired end-to-end: 1:1 DMs, group messages
  (`tryRouteGroupRumor` + the live DM subscription), incoming
  payments and zaps (`PaymentNotifier`)
* Cross-platform background detect-and-ping via `expo-background-task`
  + `backgroundSyncService` (WorkManager / BGTaskScheduler), registered
  in `backgroundTask.ts`
* Suppress-when-foreground: no notification for the thread the user is
  currently viewing (`setActiveThread` / `isThreadActivelyViewed`)
* Lock-screen-content opt-in toggle in Settings (SecurityScreen)
* Notification-tap routing in `App.tsx` (`navigateFromNotification`)
* Android manifest permissions scaffolded for a future foreground
  service via `plugins/withForegroundService.js`
* iOS Info.plist `UIBackgroundModes`
* This document

Deferred to follow-up PRs:

* Kotlin `NostrRelayService` foreground service + manifest `<service>`
  entry + `BootReceiver` — only needed for *realtime* (vs ~15-min)
  Android delivery; the background-task path ships without it
* In-app per-channel mute UI (system Settings is the mute path for now)
* Notification grouping (multiple from same sender → one stack)
