= Solution Design

== On-Chain Wallet Support & Transfer (Issues #10, #11)

=== Overview

Adds on-chain Bitcoin wallet support (watch-only via xpub) alongside existing NWC Lightning wallets, plus a Transfer feature for moving funds between wallets and Boltz submarine swaps for Lightning-to-on-chain payments.

=== Bitcoin Crypto Library Stack

The same libraries used by BlueWallet (1M+ users, React Native):

[cols="1,1,2", options="header"]
|===
| Package | Version | Purpose

| https://github.com/bitcoinjs/bitcoinjs-lib[bitcoinjs-lib] | 7.x | Core Bitcoin library — address generation, transaction building
| https://github.com/bitcoinjs/bip32[bip32] | 5.x | BIP-32 HD wallet key derivation from xpub
| https://github.com/bitcoinerlab/secp256k1[`@bitcoinerlab/secp256k1`] | latest | Pure JS secp256k1 (replaces WASM-based `tiny-secp256k1` which doesn't work in React Native)
| https://github.com/paulmillr/noble-curves[`@noble/curves`] | 2.x | Audited elliptic curve cryptography (dependency of @bitcoinerlab/secp256k1)
| https://github.com/paulmillr/noble-hashes[`@noble/hashes`] | 2.x | Audited hash functions (dependency of @noble/curves)
| https://github.com/feross/buffer[`buffer`] | latest | Node.js Buffer polyfill for React Native
|===

**Why these libraries?**

- `tiny-secp256k1` uses WebAssembly which is incompatible with React Native's Hermes engine
- `@bitcoinerlab/secp256k1` provides the same interface using `@noble/secp256k1` (pure JavaScript)
- All address derivation happens locally on-device — no third-party API needed for crypto operations
- Balance and transaction lookups use mempool.space REST API (configurable, self-hostable)

**Metro configuration:** `@noble/curves` and `@noble/hashes` use ES module `exports` maps with `.js` extensions. Metro requires `unstable_enablePackageExports: true` in `metro.config.js` to resolve these correctly.

=== Architecture

[source]
----
WalletContext.tsx
  ├─ NWC wallets → nwcService.ts (existing)
  └─ On-chain wallets → onchainService.ts (new)
       ├─ bitcoinjs-lib + bip32 — local address derivation
       └─ mempool.space API — balance & transaction lookups

HomeScreen.tsx
  ├─ [Receive] → ReceiveSheet (Lightning invoice OR Bitcoin address QR)
  ├─ [Transfer] → TransferSheet (wallet-to-wallet)
  └─ [Send] → SendSheet (Lightning invoice, address, OR Bitcoin address via Boltz)
----

type OnchainImportMethod = 'xpub' | 'mnemonic' | 'generated';
----

On-chain wallets are currently watch-only (xpub import). The type system is designed for future mnemonic import and wallet generation.

=== Transfer Logic

[cols="1,1,1", options="header"]
|===
| Source | Destination | Method

| NWC Lightning | NWC Lightning | Generate invoice on dest, pay from source. Instant.
| NWC Lightning | On-chain | Boltz reverse submarine swap (see below). ~10-60 min.
| On-chain | Any | Not supported (watch-only) — future: enable with mnemonic import
|===

=== Boltz Reverse Submarine Swap (Lightning → On-chain)

Uses https://docs.boltz.exchange[Boltz Exchange] v2 API for trustless Lightning-to-on-chain transfers.

**Flow:**

1. Generate random 32-byte preimage + SHA-256 hash
2. Generate temporary secp256k1 claim keypair
3. `POST /v2/swap/reverse` with `preimageHash`, `claimPublicKey`, `claimAddress`, `invoiceAmount`
4. Boltz returns a Lightning invoice + swap tree (Taproot HTLC scripts)
5. Pay the Lightning invoice via NWC
6. Poll `GET /v2/swap/reverse/{id}` until status = `transaction.mempool`
7. Construct a **script-path claim transaction** (Taproot spend):
   - Witness: `<schnorr_signature> <preimage> <claim_script> <control_block>`
   - Output: destination on-chain address
8. Broadcast claim tx via mempool.space API
9. Funds arrive at destination after ~1 block confirmation (~10 min)

**Why script-path (not cooperative key-path)?**

Boltz v2 Taproot swaps support two claim methods:

- **Key-path (cooperative):** Requires MuSig2 nonce exchange between client and Boltz. Smaller on-chain footprint, better privacy (looks like a normal tx). BUT MuSig2 needs WASM or native code — React Native's Hermes engine can't run it without a WebView workaround.
- **Script-path (non-cooperative):** Uses standard Schnorr signature + preimage reveal. Works with pure JS (`@noble/curves`). Slightly larger on-chain tx but settles just as fast (~10 min).

We use script-path because it works with our existing pure JS crypto stack. Future: upgrade to cooperative key-path using `@scure/btc-signer` MuSig2 if Hermes adds WASM support.

**Libraries:**

[cols="1,2", options="header"]
|===
| Package | Purpose

| https://github.com/nicoll-douglas/scure-btc-signer[`@scure/btc-signer`] | Taproot transaction construction and script-path spending
| https://github.com/bitcoinerlab/secp256k1[`@bitcoinerlab/secp256k1`] | Schnorr signing (BIP-340) via @noble/curves
| https://github.com/nicoll-douglas/noble-curves[`@noble/curves`] | Elliptic curve cryptography (secp256k1 Schnorr)
|===

**Fallback:** If the claim tx fails to broadcast, Boltz's Protocol 11 automatically sends funds to the `claimAddress` after the timeout period (~24 hours).

**Current status:** The full Boltz reverse swap flow is implemented: swap creation, invoice payment, lockup detection, and script-path claim transaction (Schnorr signature + preimage reveal). The claim tx is broadcast via mempool.space and funds arrive after ~1 block confirmation (~10 min). The swap state (preimage + claim private key) needs to be persisted to SecureStore so it survives app restarts — tracked in https://github.com/BenGWeeks/lightning-piggy-mobile/issues/38[#38]. The claim keypair and preimage should be viewable in the transaction detail bottom sheet for manual recovery or Boltz support contact.

=== NWC Relay Compatibility

The LNbits Nostrclient relay proxy does not send NIP-01 OK responses after receiving EVENT messages (see https://github.com/lnbits/nostrclient/issues/52[lnbits/nostrclient#52]). This causes `nostr-tools` `SimplePool.publish()` to timeout, breaking all NWC operations.

**Workaround in `nwcService.ts`:** The relay pool's `publish` method is monkey-patched during `connect()` to resolve immediately (fire-and-forget) instead of waiting for OK. The NWC response comes back via the subscription, not the OK acknowledgement.

**Proper fix:** PR https://github.com/lnbits/nostrclient/pull/68[lnbits/nostrclient#68] adds NIP-01 OK and CLOSED responses to Nostrclient. Once merged, the monkey-patch can be removed.

Additionally, `nwcService.ts` includes auto-reconnect logic: if a publish fails with "failed to publish" (relay disconnected), it creates a fresh `NostrWebLNProvider` and retries. The wallet is marked as "Connected" after `enable()` succeeds, not after `getBalance()`, so slow relay responses don't show the wallet as disconnected.

=== Visual Differentiation

- **NWC wallets**: Green/red status dot + "Connected"/"Disconnected"
- **On-chain wallets**: Chain icon (⛓) + "On-chain" label, no connection status

=== Future: Hot Wallet Support

When mnemonic import is added, the following will be implemented in `onchainService.ts`:

- `buildTransaction()` — construct PSBTs using bitcoinjs-lib
- `broadcastTransaction()` — sign locally and broadcast via mempool.space API
- Transfer from on-chain wallets (currently disabled for watch-only)

The crypto library stack (bitcoinjs-lib, bip32, @bitcoinerlab/secp256k1) already supports all of this.

= Nostr Integration

== Overview

Lightning Piggy is a Lightning wallet app for kids built with React Native (Expo). The Nostr integration adds social features: identity, friends, profile management, and zaps. Nostr is optional — the app works as a standalone wallet without it.

== Architecture

[source]
----
App.tsx
  └─ NostrProvider (context)
       ├─ nostrService.ts    — stateless relay I/O (read/write)
       ├─ amberService.ts    — Android Amber signer bridge
       └─ imageUploadService.ts — nostr.build image uploads
----

=== Key Files

[cols="1,2", options="header"]
|===
| File | Purpose

| `src/contexts/NostrContext.tsx` | React context providing login state, profile, contacts, and all Nostr actions
| `src/services/nostrService.ts` | Stateless functions: relay connections, profile/contact fetching, event creation and signing
| `src/services/amberService.ts` | Android Amber signer integration via NIP-55 URIs
| `src/services/imageUploadService.ts` | Image upload to nostr.build API v2
| `src/screens/FriendsScreen.tsx` | Friends tab with FlashList, search, alphabet sidebar
| `src/screens/AccountScreen.tsx` | Profile display, settings, login/logout
| `src/components/NostrLoginSheet.tsx` | Login (nsec/Amber) and account creation flows
| `src/components/EditProfileSheet.tsx` | Profile editing with image picker + nostr.build upload
| `src/components/ContactProfileSheet.tsx` | Contact profile card with follow/unfollow
| `src/components/AddFriendSheet.tsx` | Add friend by npub paste or QR scan
|===

== Authentication

Two sign-in methods are supported:

=== nsec (Private Key)

1. User enters `nsec1...` key in the login sheet
2. Key is decoded via `nostr-tools/nip19` to derive the pubkey
3. nsec stored in `expo-secure-store` (encrypted device storage)
4. All event signing done locally with `nostr-tools/pure.finalizeEvent()`

=== Amber Signer (Android only)

1. App sends an intent to the Amber app via `Linking.openURL()`
2. Amber returns the pubkey (no private key ever leaves Amber)
3. Pubkey stored in `expo-secure-store`
4. Event signing delegated to Amber via NIP-55 intents

=== Auto-Login on Startup

On app launch, `NostrContext` checks `expo-secure-store` for stored credentials:

- **nsec path**: reads stored nsec, derives pubkey, sets `isLoggedIn = true`
- **Amber path**: reads stored pubkey, sets `isLoggedIn = true`

No re-authentication with Amber is needed on startup. The pubkey alone is sufficient to restore the logged-in state and fetch all public data.

=== Account Creation

New users can generate a Nostr identity directly in the app:

1. `nostr-tools/pure.generateSecretKey()` creates a new keypair
2. User enters a display name
3. nsec is stored and the user is logged in
4. A kind 0 (metadata) event is published with the display name
5. User is shown their nsec with a copy button and backup warning

== Data Flow

=== What Requires Only the Pubkey (No Signing)

All read operations use just the pubkey — no private key or Amber interaction:

- Fetch profile (kind 0)
- Fetch contact/follow list (kind 3)
- Fetch relay list (kind 10002 / NIP-65)
- Fetch profiles for all contacts (batch)
- Load cached contacts from AsyncStorage

=== What Requires Signing (nsec or Amber)

Write operations that publish events to relays:

[cols="1,1,1", options="header"]
|===
| Action | Event Kind | Description

| Edit profile | kind 0 | Publishes updated metadata (name, picture, banner, lud16, about)
| Follow contact | kind 3 | Publishes updated contact list with new pubkey added
| Unfollow contact | kind 3 | Publishes updated contact list with pubkey removed
| Zap request | kind 9734 | Signs a zap request for NIP-57 Lightning zaps
|===

Events are published to the user's **write relays** (from NIP-65), falling back to `DEFAULT_RELAYS` if none are configured.

=== NIP-65 User Relays

On login, the app fetches the user's kind 10002 event to discover their preferred read/write relays. These are merged with the defaults for all subsequent operations.

== Performance

Loading 500+ contacts with profiles requires careful optimization:

=== Caching

- Contact list and profiles cached in `AsyncStorage` with 24-hour TTL
- On startup, cached contacts load in ~40ms (no network)
- Fresh data fetched in background via `InteractionManager.runAfterInteractions`

=== Rendering

- `@shopify/flash-list` replaces FlatList for cell recycling (5-10x faster)
- `React.startTransition()` wraps all `setContacts` calls for non-blocking renders
- `freezeOnBlur: true` on tab navigator prevents re-renders on hidden tabs
- Context value memoized with `useMemo` to prevent unnecessary consumer re-renders

=== Profile Fetching

- Batches of 50 pubkeys, 3 concurrent batches, 12s timeout per batch
- 50ms yield between batch rounds for UI responsiveness
- `onBatch` callback updates UI incrementally as profiles arrive
- `expo-image` with `cachePolicy="disk"` for profile pictures

== Image Upload

Profile pictures and banners are uploaded via the nostr.build API:

- **Endpoint**: `POST https://nostr.build/api/v2/upload/files`
- **Method**: multipart/form-data with `file` field
- **Auth**: Anonymous (no NIP-98 required)
- **Response**: NIP-94 event with `url` tag containing the hosted image URL
- **Image picker**: `expo-image-picker` with 1:1 crop for avatars, 3:1 for banners

== Deep Linking

- Contact profiles link to Primal via `https://primal.net/p/{npub}`
- `nostr:` URI scheme attempted first but Primal doesn't register for it on Android
- App registers `lightningpiggy://` scheme for Amber callbacks
