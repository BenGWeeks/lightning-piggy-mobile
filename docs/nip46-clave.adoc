= NIP-46 Signer (Nostr Connect / Clave / Aegis / nsec.app)
:toc:
:toclevels: 2

== Overview

Lightning Piggy supports three Nostr signers:

[cols="1,1,2"]
|===
| Signer | Platform | Transport

| `nsec`
| All
| Local ‚Äî the user's nsec is decrypted in app memory.

| `amber` (NIP-55)
| Android only
| Android Intent IPC. Requires the Amber app installed locally.

| `nip46` (NIP-46 / Nostr Connect)
| All (iOS, Android, web)
| NIP-04 encrypted DMs over a Nostr relay. Bunker can run anywhere reachable on a relay.
|===

NIP-46 closes the iOS gap ‚Äî Doc's https://testflight.apple.com/join/5Mx5AZx7[Clave] is the canonical iOS bunker. It also lets Android users pair with desktop / web bunkers (https://nsec.app[nsec.app]) for hardware-isolated key storage without installing Amber.

See issue #283 for the design comparison and parity table.

== Pairing flow

The pairing handshake is one-shot per device. Once paired, the connection is persisted to `expo-secure-store` and auto-restored on cold start ‚Äî the user only ever scans a QR once per app install.

. From the **Account ‚Üí Connect Nostr** sheet, tap **Use NIP-46 Signer (Clave / nsec.app)**.
. Lightning Piggy generates a per-app keypair (never the user's real nsec) and renders a `nostrconnect://` URI as a QR code.
. The user opens their bunker app (Clave / Aegis / nsec.app) and scans the QR.
. The bunker prompts the user to approve the pairing (showing the requested perms: `sign_event`, `nip04_encrypt`, `nip04_decrypt`, `nip44_encrypt`, `nip44_decrypt`).
. On approval, the bunker publishes a NIP-04 encrypted `connect` ack to the relay specified in the URI.
. Lightning Piggy receives the ack, persists the connection, and the user is logged in.

If the bunker doesn't ack within 120s, the pairing screen shows "Pairing took too long ‚Äî try again". The user can retap the button to start over.

== Per-call behaviour

Every signing operation (sign event, NIP-04 encrypt/decrypt, NIP-44 encrypt/decrypt) becomes a relay round-trip:

. Lightning Piggy publishes an encrypted `kind:24133` request to the bunker via the paired relay.
. The bunker (optionally) prompts the user to approve the call.
. The bunker publishes an encrypted response back via the same relay.
. Lightning Piggy decrypts the response.

Per-call latency is **~200-1500 ms** depending on the relay. Compare with Amber's ~10-50 ms IPC fast-path.

Where this matters in practice:

* **DM send** (1 NIP-04 encrypt + 1 sign) ‚Äî 0.4-3 s extra vs nsec.
* **Group send to N members** (sequential N √ó NIP-44 encrypt + N √ó sign) ‚Äî 5-15 s for a 5-member group.
* **Inbox refresh of K NIP-17 wraps** ‚Äî K √ó per-call latency.

Lightning Piggy renders a spinner / progress UI during these flows so the user knows the app is working.

== The silent-decrypt batch fast-path trade-off

Amber has a `nip44_decrypt_batch` ContentResolver fast-path ‚Äî one IPC for N decrypts, no per-call user prompts. NIP-46 has no equivalent in the spec today. Two options were considered:

. Spec-extend with a batched bunker request and ship it across Clave / nsec.app / Aegis.
. Take the slow-but-correct path: per-request round-trip, render a spinner.

This PR takes option 2. Inbox refresh on cold cache will show a "decrypting‚Ä¶" spinner while the wraps catch up. Two on-device short-circuits mean subsequent refreshes only pay the round-trip cost for newly-arrived wraps, so the steady-state UX is the same as nsec: decrypted rumors persist as rows in the encrypted SQLCipher store (`src/services/dmDb`), so a wrap whose id is already stored is never re-decrypted; and a per-signer negative skip-set (`nip46_nip17_skip_v1`, persisted via `expo-file-system`) records wrap ids that decrypted to a non-inbox rumor (group message / non-followed sender) so the loop skips them too. The legacy plaintext `*_nip17_cache_v1` AsyncStorage caches were retired in #848.

Bunker-side batching is filed as a follow-up. See `src/services/nostrConnectService.ts` `requestNip44DecryptSilent` for the deliberate "throw to surface the trade-off" sentinel.

== Bunker compatibility

The `nostrconnect://` URI we generate matches the format defined in the NIP-46 spec, section "Nostr Connect URIs":

----
nostrconnect://<client-pub>?relay=wss://relay.nsec.app&secret=<32hex>&perms=sign_event,nip04_encrypt,nip04_decrypt,nip44_encrypt,nip44_decrypt&name=Lightning%20Piggy
----

Bunkers known to consume this format (no app-side testing, just spec compliance):

* **Clave** (iOS, https://testflight.apple.com/join/5Mx5AZx7[TestFlight]) ‚Äî Doc's signer, the primary target for this PR.
* **Aegis** (iOS / Android, https://github.com/ZharlieW/Aegis[GitHub])
* **nsec.app** (web, https://nsec.app[]) ‚Äî paste the URI as text.
* **Nowser** (Android, https://github.com/haorendashu/nowser[GitHub])
* **Nostrum**

The relay default is `wss://relay.nsec.app`. Bunkers that subscribe to other relays will need URI customisation ‚Äî that's a follow-up. Most bunkers either default to the same relay or watch a wide set including this one.

== Storage

[cols="1,2"]
|===
| Key | Contents

| `nostr_signer_type` (SecureStore)
| `'nip46'`

| `nostr_pubkey` (SecureStore)
| The user's hex pubkey (as reported by the bunker's `get_public_key`).

| `nostr_nip46_connection` (SecureStore)
| JSON `Nip46Connection` ‚Äî `remoteSignerPubkey`, `userPubkey`, `relays`, `clientSecretKeyHex`, `perms`. **Sensitive** ‚Äî anyone with this can impersonate the app session against the bunker.

| `nip46_nip17_skip_v1` (AsyncStorage)
| Skip-set of NIP-17 wrap ids that failed to unwrap (e.g. bunker denied `nip44_decrypt`), so a refresh doesn't re-attempt them every pass. Cleared on logout. Unlike the `nsec_`/`amber_` paths there is no `nip46_nip17_cache_v1` ‚Äî decrypted rumors persist in the encrypted local DM store, not an AsyncStorage wrap-id ‚Üí rumor cache.
|===
