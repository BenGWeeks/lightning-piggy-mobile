# Security source audit — rust-nostr @ `nostr-sdk-kmp-android:0.44.3`

**Verdict: SAFE TO PIN** — with one low-severity spec-deviation to track (non-constant-time HMAC comparison in NIP-44 decrypt).

The source correctly wires audited, mainstream crypto primitives (bitcoin-core `secp256k1`, RustCrypto `chacha20`/`hmac`/`sha2`). No hand-rolled curve math. NIP-44 conversation-key derivation, nonce handling, padding, MAC-then-decrypt ordering, and version handling all match the spec. NIP-59 gift-wrap unwrap performs the seal-signature check AND the rumor-pubkey == seal-signer check that defeats the classic impersonation pitfall. Maven artifact PGP chain traces to Yuki Kishimoto's long-standing, GitHub-registered key. No RUSTSEC/OSV advisory touches any crate on the NIP-44 / schnorr / NIP-59 code path.

## 0. Provenance of what was audited

The KMP artifact is **not** built from the `rust-nostr/nostr` repo directly. It is generated by a separate wrapper repo:

- **`rust-nostr/nostr-sdk-ffi`** — uniffi + `gobley-uniffi-bindgen` KMP wrapper. Audited at branch `v0.44`, **commit `3e2170c8b6fe287af46827f7d61c1115137b3b8f`** ("kmp: bump to 0.44.3"). The KMP artifact version `0.44.3` is a wrapper-only bump; there is no `v0.44.3` git tag (tags stop at `v0.44.2`) — the `.pom` `<scm>` points at `nostr-sdk-ffi.git`, and the version is set in the KMP gradle build, confirmed by the commit message.
- The wrapper's `Cargo.lock` pins the actual crypto crate **`nostr` = 0.44.2** (checksum `3aa5e3b6…`, from crates.io). This is the code that implements NIP-44/schnorr/NIP-59. I downloaded that exact crate from crates.io; its sha256 matches the lockfile checksum byte-for-byte.
- The nip44/nip59/util/event-id source in that crate is **identical** to `github.com/rust-nostr/nostr@v0.44` (verified `diff -q` on all six relevant files → MATCH).

KMP build pipeline (`.github/workflows/publish-kmp.yml`): builds `libnostr_sdk_ffi.so` per-target, runs `gobley-uniffi-bindgen` to emit Kotlin, assembles the AAR, publishes to Maven Central with GPG signing. The published AAR (`nostr-sdk-kmp-android-0.44.3.aar`, 17 MB) contains `classes.jar` + `jni/{arm64-v8a,armeabi-v7a,x86,x86_64}/libnostr_sdk_ffi.so` — i.e. the Rust is compiled to native `.so` and JNA-bridged, exactly as expected.

FFI wrappers are thin pass-throughs (no re-implementation):
- `src/protocol/nips/nip44.rs` → calls `nostr::nips::nip44::{encrypt,decrypt}` directly.
- `src/protocol/nips/nip59.rs` → `UnwrappedGift::from_gift_wrap` delegates to `nip59::UnwrappedGift::from_gift_wrap`.
- `src/protocol/event/mod.rs:77` `verify()` → `self.inner.verify()`.

## 2. NIP-44 implementation review — `nostr-0.44.2/src/nips/nip44/`

Reviewed against the spec (`nips/44.md`). Findings:

- **Conversation key derivation** — `v2.rs:139` `ConversationKey::derive`: computes ECDH via `util::generate_shared_key` (`util/mod.rs:36`) which normalises the x-only pubkey to even parity and takes `ecdh::shared_secret_point(...)[..32]` (the x-coordinate) — correct per spec ("shared_x"). Then `hkdf::extract(b"nip44-v2", &shared_key)` → HKDF-extract with salt `nip44-v2` and IKM = shared_x. **Correct** (`util/hkdf.rs:15`: `HMAC(salt, IKM)`).
- **Message-key expansion** — `v2.rs:296` HKDF-expand with `info = nonce`, `L = 76`, sliced into chacha_key[0..32] / chacha_nonce[32..44] / hmac_key[44..76]. Matches spec byte ranges exactly (`mod.rs:28-35`).
- **Per-message nonce** — `v2.rs:196-203`: 32 random bytes from `OsRng` on every encrypt; never derived from message content, never reused. Correct. (An override path exists but is only reachable by the test harness via a private fn.)
- **Padding** — `pad()` (`v2.rs:300`) prepends `u16` BE length, zero-pads to `calc_padding(len)`; min 32 bytes, power-of-two chunking. Matches spec `calc_padded_len`; validated by the bundled `calc_padded_len` vectors.
- **MAC-then-decrypt ordering** — `decrypt_to_bytes` (`v2.rs:233`) computes HMAC over `nonce || ciphertext` and **returns `InvalidHmac` before** running ChaCha20 (`v2.rs:253-259` guard precedes cipher at `:262`). Correct — no decrypt-on-bad-MAC.
- **Padding-oracle hardening** — after decrypt, validates declared length, non-empty, and that total buffer length equals `2 + calc_padding(unpadded_len)` (`v2.rs:271-286`). Good.
- **Version byte** — `mod.rs:170` reads `payload[0]`, `TryFrom<u8>` rejects anything ≠ 0x02 with `UnknownVersion` (`mod.rs:94-103`). Version byte is intentionally excluded from the MAC (spec MACs `aad(nonce) || ciphertext` only) — matches spec.
- **⚠ CONCERN (low severity): non-constant-time MAC comparison.** `v2.rs:257`: `if mac != calculated_mac.as_slice()` uses Rust's stdlib slice equality, which short-circuits on the first differing byte — i.e. **variable-time**. The NIP-44 spec (44.md line 125) explicitly requires "constant-time comparison". `subtle::ConstantTimeEq` is in the dependency tree but is not used here. Practical exploitability is low (remote timing attack against per-message HMAC keys over a noisy relay→wallet path, with no obvious timing oracle reflected to an attacker in a wallet context), but this is a genuine deviation from the audited reference. **Recommend** filing upstream / tracking; a one-line switch to `subtle` fixes it.

**Test vectors:** the crate bundles `nip44.vectors.json` and runs the official paulmillr vectors (`v2.rs:340-648`: `get_conversation_key`, `calc_padded_len`, `encrypt_decrypt`, invalid-decrypt with expected error codes). I diffed the bundled vectors against the current upstream `paulmillr/nip44` file: the **`encrypt_decrypt` round-trip set is byte-identical**. The bundled snapshot has fewer `get_conversation_key` cases (3 vs 35 upstream) — it is an older snapshot, not a mismatch; all shared vectors agree. The MAC-comparison concern does not affect vector correctness (vectors test valid MACs). **Follow-up:** run `cargo test -p nostr nip44` against the latest upstream vectors to close the loop (I verified statically, did not compile — secp256k1 C build not run here).

## 3. Schnorr / event verification

- **Signing** — `key/mod.rs:238` `sign_schnorr` → `secp.sign_schnorr_with_rng(message, keypair, rng)` (bitcoin-core secp256k1). No hand-rolled math.
- **Verification** — `event/mod.rs:203` `verify_signature_with_ctx`: `Message::from_digest(self.id.to_bytes())` then `secp.verify_schnorr(&self.sig, &message, &public_key)` with the x-only pubkey. BIP-340 via the C library.
- **Full verify** — `event/mod.rs:165` `verify()` checks **both** `verify_id()` and the signature; a bad id short-circuits before signature check.
- **Event id** — `event/id.rs:42` `EventId::new`: `sha256(json!([0, pubkey, created_at, kind, tags, content]).to_string())` — the NIP-01 canonical serialization. Correct.

No red flags.

## 4. NIP-59 gift-wrap unwrap (M2 path) — `src/nips/nip59.rs`

`UnwrappedGift::from_gift_wrap_with_ctx` (`nip59.rs:89`):
1. Kind must be `GiftWrap` else `NotGiftWrap` (`:99`).
2. NIP-44-decrypt the wrap content → parse **seal**, then **`seal.verify_with_ctx(secp)?`** — verifies the seal's schnorr signature against the seal's own pubkey (`:108`). (The outer wrap is signed by a throwaway ephemeral key and is intentionally not authenticated against any sender — correct expectation.)
3. NIP-44-decrypt the seal content → parse **rumor** (`UnsignedEvent`).
4. **`if rumor.pubkey != seal.pubkey { return Err(SenderMismatch) }`** (`:115`) — this is the anti-impersonation check. The reported `sender` is taken from `seal.pubkey`, not from the rumor. This matches Lightning Piggy's JS unwrap requirement.

There is an explicit regression test for the spoofing case (`nip59.rs:205 test_sender_mismatch`: a rumor lying about its pubkey but wrapped by a different key is rejected with `SenderMismatch`). Wrap-side `gift_wrap_from_seal` (`event/builder.rs:1474`) uses a fresh `Keys::generate()` (OsRng) ephemeral key per wrap — correct.

**No impersonation pitfall.** The native path enforces the same check as the JS path.

## 6. Relay-pool structure pass (brief, M2)

`nostr-relay-pool 0.44.0` (checksum verified against lockfile). On event ingestion (`relay/inner.rs:1219-1231`), events are `verify()`-checked (id + signature) **before** being saved, and invalid events are not stored. A verification cache (`shared.rs:140`) avoids re-verifying. **Minor note:** the cache keys on a **64-bit `DefaultHasher` hash of the event id**, not the full 32-byte id — a hash collision could let an unverified event be treated as already-verified. `DefaultHasher` (SipHash) is randomly seeded per process, so this is not remotely steerable and is low risk, but it is a theoretical soundness gap worth an upstream note. Nothing else alarming at structure level.
