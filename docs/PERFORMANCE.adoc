= Performance measurement

This guide covers how to measure Lightning Piggy's runtime perf, what the
output means, and how to feed it to Stev.ie (the project's perf agent —
`.claude/agents/stevie.md`).

Scope: **dev-time only.** Lightning Piggy targets privacy-centric users
(GrapheneOS / degoogled Android) so we never ship commercial telemetry —
no Sentry, no Datadog, no Firebase Performance. Everything here runs on
your own device and your own machine.

== Quick start

[source,bash]
----
# 1. Set the emulator GPS so location-aware screens have a fix.
#    Longitude first (gotcha), latitude second:
adb emu geo fix 0.0589 52.2964   # Longstanton, near a cached merchant

# 2. Run the Explore cold-start measurement, 5 samples:
bash scripts/perf-explore-cold-start.sh 5

# 3. With a Perfetto trace on run 1 (if you want to see *where* the time goes):
PERFETTO=1 bash scripts/perf-explore-cold-start.sh 5
----

== What the scripts report

=== Wall-clock distribution

Every `scripts/perf-*.sh` now emits a six-line block via the shared
`scripts/lib/perf-stats.sh` helper:

[source]
----
Cold-start wall-clock (n=5):
  min   4830 ms
  p50   5210 ms
  p95   6240 ms
  p99   6890 ms
  max   6890 ms
  mean  5430 ms
----

**Always read p99/max, not mean.** A mean of "24 s" can hide a 38 s
outlier — and outliers are the user-visible regression. Means are
provided for trend-tracking, but the actionable metric is p99.

=== Frame jank (per run)

Each run prints a `gfxinfo:` line right after its wall-clock:

[source]
----
run 1: 5230 ms — gfxinfo: total=387 janky=12 (3.1%) p50=8ms p95=15ms p99=24ms
----

[cols="1,3", options="header"]
|===
| Field | Meaning

| `total`
| Frames rendered during the run. Higher = more on-screen activity.

| `janky` (count and %)
| Frames that took >16.67 ms (60 Hz) to commit. **>5%** is noticeable to
the user; **>10%** is bad.

| `p50`
| Median frame time. Should be well under 16 ms on a healthy screen.

| `p95` / `p99`
| The tail. A p95 of 60 ms means 1 in 20 frames stuttered for ~4 frames
worth — visible jank.
|===

=== Perfetto trace (optional, `PERFETTO=1`)

When `PERFETTO=1` is set, the first run captures a 40 s system trace
covering the scheduler, gfx + view + input atrace categories, and the
target app's atrace events. Output:

[source]
----
perfetto: pulled to /tmp/explore-cold-start-trace-194233.pftrace — open at https://ui.perfetto.dev (drag-drop the file)
----

Drop the `.pftrace` on https://ui.perfetto.dev. You can then:

- See per-thread CPU usage during the freeze (look for the JS thread)
- Spot the long `setState` commit blocks
- Confirm whether jank lives in the JS thread, the native bridge, or GC

This is the canonical way to diagnose a freeze — wall-clock alone never
names the culprit.

== Methodology

Stev.ie owns the deeper methodology — see `.claude/agents/stevie.md`
for the 12-category audit (JS-thread blocking, per-event setState,
context re-render cascades, subscription lifecycle, etc.). The short
version:

1. **Measure** before changing anything. Wall-clock + gfxinfo + a
   Perfetto trace = three independent angles on the same problem.
2. **Audit** the code path implicated. Static analysis often beats
   adding more instrumentation — every category in Stev.ie's
   methodology has a "grep for it" form.
3. **Fix one thing at a time.** Don't ship a 5-issue bundle. Measure
   after each.
4. **Verify** with a fresh measurement. If p99 didn't move, the fix
   didn't move the needle.

== Adding a new perf script

Use this skeleton — it inherits the stats + gfxinfo + Perfetto helpers:

[source,bash]
----
#!/usr/bin/env bash
set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
# shellcheck source=lib/perf-stats.sh
source "$SCRIPT_DIR/lib/perf-stats.sh"

RUNS=${1:-3}
DEVICE=${PIGGY_DEVICE:-emulator-5554}
APP_ID=${PIGGY_APP_ID:-com.lightningpiggy.app.dev}

totals=()
for i in $(seq 1 "$RUNS"); do
  perf_gfxinfo_reset "$DEVICE" "$APP_ID"
  [ "$i" = "1" ] && perf_perfetto_start "$DEVICE" 40000 "$APP_ID"

  t1=$(date +%s%3N)
  # … your scenario here: launchApp, tap, wait, etc …
  t2=$(date +%s%3N)
  totals+=("$((t2 - t1))")
  printf '  run %d: %d ms — ' "$i" "$((t2 - t1))"
  perf_gfxinfo_sample "$DEVICE" "$APP_ID"

  [ "$i" = "1" ] && perf_perfetto_stop "$DEVICE" "/tmp/my-trace-$(date +%H%M%S).pftrace"
done
perf_stats_report "My measurement" "${totals[@]}"
----

== Setting up the emulator for accurate GPS

`expo-location` reads through Android's AOSP `LocationManager` — no
Google Play Services needed (Lightning Piggy intentionally avoids the
Fused Location Provider; see memory `feedback_no_google_location_services`).

If the app sits in "Locating you…" forever:

[source,bash]
----
# 1. Confirm the location service is registered (it should be on any
#    standard AVD image — `default` or `google_apis`; `google_apis_playstore`
#    is NOT required).
adb shell service list | grep location
# Expect: location: [android.location.ILocationManager]

# 2. Push a fix. NB: longitude first, then latitude.
adb emu geo fix 0.0589 52.2964   # Longstanton

# 3. Verify the app picks it up (re-launch may be needed if location
#    permission hadn't been granted).
adb shell pm grant com.lightningpiggy.app.dev android.permission.ACCESS_FINE_LOCATION
adb shell pm grant com.lightningpiggy.app.dev android.permission.ACCESS_COARSE_LOCATION
----

The `EXPO_PUBLIC_DEV_LAT` / `EXPO_PUBLIC_DEV_LON` env-pin workaround
that used to live in `src/utils/devLocation.ts` has been removed
(merged in PR #597 / issue #604) — we now rely on a properly configured
AVD instead. This means dev now exercises the same GPS code path as
production, so location-related perf measurements actually represent
real behaviour.
