= Troubleshooting

[cols="1,2", options="header"]
|===
| Problem | Solution

| Boltz reverse swap claim broadcast fails with "mandatory-script-verify-flag-failed (Witness program hash mismatch)"
| The Taproot internal key for Boltz v2 swaps is a MuSig2 aggregate of Boltz's key and the user's key — in a specific order. Per Boltz docs, **"Boltz's public key always coming first"**: for reverse swaps that means `keyAggregate([refundPubKey, claimPubKey])` (Boltz's refund key first, user's claim key second; **not sorted, not the other way round, not just refundPubKey alone**). Keys must be 33-byte compressed, aggregated via `@scure/btc-signer`'s `keyAggregate` + `keyAggExport`, then used as the x-only internal key when computing the Taproot tweak. See `boltzService.ts` `claimSwap()` and the Boltz docs at https://api.docs.boltz.exchange/claiming-swaps.html. Confirmed against the chain by computing `p2tr(internalKey, [claimLeaf, refundLeaf], undefined, true)` from `@scure/btc-signer` and matching its `.script` to the on-chain lockup scriptPubKey.

| Bottom sheet doesn't slide up when keyboard opens on Android
| All of these are required: (1) Use `BottomSheetTextInput` from `@gorhom/bottom-sheet` instead of React Native's `TextInput`. (2) Use `BottomSheetScrollView` (not `BottomSheetView`) as the content wrapper. (3) Set `keyboardBehavior="interactive"` (or `"extend"` if the sheet has multiple snap points and you want the IME to push it to the highest snap — see `GifPickerSheet`), plus `keyboardBlurBehavior="restore"` and `android_keyboardInputMode="adjustResize"` on the `BottomSheetModal`. (4) Ensure `android:windowSoftInputMode="adjustResize"` in AndroidManifest (via `plugins/withAdjustResize.js`). (5) Add a `Keyboard.addListener` to track keyboard height and set dynamic `paddingBottom` on the `contentContainerStyle` (e.g. `paddingBottom: keyboardHeight > 0 ? keyboardHeight + 80 : 40`). (6) Do NOT wrap content in `TouchableWithoutFeedback` with `Keyboard.dismiss` — this interferes with the sheet's keyboard handling. See `NostrLoginSheet.tsx` as the reference implementation.

| AVD soft keyboard never appears (focus an input → no IME, or IME blinks then dismisses)
| The Android emulator suppresses the on-screen IME when it detects a "hardware keyboard" (the AVD reports the host keyboard as attached). On API 35+ a second gate kicks in: `stylus_handwriting_enabled` defaults to `1`, which routes some focus events into the stylus-handwriting service instead of bringing up Gboard. Both gates regress on AVD restart unless re-applied. +
*Fix (apply BOTH; `setprop`-style settings persist across `am force-stop` but not across emulator boot):* +
[source,bash]
----
adb shell settings put secure show_ime_with_hard_keyboard 1
adb shell settings put secure stylus_handwriting_enabled 0
adb shell ime reset
adb shell ime enable com.google.android.inputmethod.latin/com.android.inputmethod.latin.LatinIME
adb shell ime set com.google.android.inputmethod.latin/com.android.inputmethod.latin.LatinIME
----
*Verify:* `adb shell dumpsys input_method | grep mInputShown` should report `mInputShown=true` once an input is focused. The audit script `scripts/keyboard-screenshot-audit.sh` polls this gate before every screenshot. +
*Lower-friction alternative:* run daily-driver dev work on **API 34, no Google Play, "Medium Phone" device profile**. API 34 doesn't ship the stylus-handwriting service so the second gate is a no-op, and the smaller GMS surface frees ~300 MB RAM (avoiding the "System UI isn't responding" stalls under memory pressure). API 36 is fine for production-build smoke testing — just expect to re-run both `settings put` commands after each cold boot.

| Maestro can't find elements behind a modal Alert dialog
| Dismiss the alert first (e.g. tap "KEEP CURRENT" or "Cancel") before asserting visibility of elements underneath.

| Maestro `inputText` produces invalid bech32 checksum on secure fields
| Add `.trim()` to the input value before processing. Maestro may append whitespace characters to secure text fields.

| Profiles not loading for some Nostr contacts
| Add more relay aggregators to `PROFILE_RELAYS` in `nostrService.ts` (e.g. `relay.primal.net`, `purplepag.es`). Use smaller batch sizes (50) with longer timeouts (12s) for `fetchProfiles`.

| App unresponsive after Nostr login (20s lockup)
| Wrap `setContacts` calls in `startTransition()`, use `InteractionManager.runAfterInteractions` to defer network I/O, add `freezeOnBlur: true` to tab navigator, and yield to event loop between profile fetch batches with `setTimeout(r, 50)`.

| `npx expo start` fails but `npm start` works
| The `start` script includes `--dev-client` which is required for custom native modules. Always use `npm start`.

| EAS local production build crashes mid-Gradle with `UncheckedIOException: Could not add entry ... to cache fileHashes.bin`
| `/tmp` is on tmpfs (default Ubuntu/Debian) capped at ~12 GB. The EAS local builder generates ~9 GB of native-lib intermediates during the multi-arch APK assembly (arm64 + armv7 + x86 + x86_64). When tmpfs hits 100 %, Gradle's incremental-cache writer throws on the next `fileHashes.bin` flush and the entire build aborts. +
*Symptom:* `tail -3 /tmp/eas-build.log` shows the IOException; `df -h /tmp` shows tmpfs at 100 %; the build PID is gone with no APK at `android/app/build/outputs/apk/release/app-release.apk`. +
*Fix:* point Gradle at disk-backed storage via `TMPDIR`: +
[source,bash]
----
mkdir -p "$HOME/eas-tmp"
TMPDIR="$HOME/eas-tmp" EXPO_NO_DOCTOR=1 \
  eas build --local --platform android --profile production --non-interactive
----
*Cleanup:* `rm -rf "$HOME/eas-tmp"` between builds when you want to reclaim disk; the next run's `mkdir -p` and Gradle will re-create everything, including hidden entries such as `.gradle` or `.expo`. The EAS-CLI checkout under `$TMPDIR/eas-build-local-nodejs/` is also safe to delete between runs. Full recipe + rationale in `docs/DEPLOYMENT.adoc` → "Local production builds".

| `eas build --local` produces an APK that fails to install with `INSTALL_FAILED_VERSION_DOWNGRADE` despite a fresh `eas build:version:set`
| `eas.json` has `cli.appVersionSource: "remote"` — so **both cloud and local builds** fetch the versionCode from EAS's server-side counter and (with `autoIncrement: true` on the production profile) bump it. The `app.config.ts` `versionCode` floor is ignored — eas-cli prints a warning saying so on every build. Three sharp edges interact: +
*1. Race between `set` and `build`.* `eas build:version:set` makes an async API write; `eas build --local` fetches+increments via a separate API call. If you start the build immediately after the set, the build's read can hit the server before the write commits, so the APK comes out as `(old counter) + 1` instead of `(new counter) + 1`. *Fix:* run `eas build:version:get --platform android --profile production` after `set` and before `build`, and only proceed once it reports the value you set. +
*2. The `set` prompt is interactive only.* Piping (`<<<`, `printf | …`) errors with "Input is required, but stdin is not readable". Use `script -qfc 'eas build:version:set --platform android --profile production' /dev/null <<<'NN'` for TTY emulation — but the prompt's pre-filled default may concatenate with your keystrokes (saw "5078" rendered when intent was "50"); always confirm the result with `:get` after. Or just run it interactively in a real shell. +
*3. The device floor is sticky.* Once you install versionCode N on a device, you cannot downgrade to versionCode < N without uninstalling (which loses all app data: wallets, Nostr login, message history). `adb install -r -d` (allow-downgrade) is a no-op for non-debuggable production APKs on Android 14+. So if dev's local production build ever overshoots (e.g. counter accidentally bumped to 5078), all future cloud releases <5078 fail to install on that one device until uninstall+reinstall. *Mitigation:* keep the EAS counter conservative and bump in single digits when forced — never bulk-set to a future-proof high number. +
*Symptom + diagnosis sequence:* +
[source,bash]
----
adb -s <serial> install -r build-*.apk
# → Failure [INSTALL_FAILED_VERSION_DOWNGRADE: Update version code 28 is older than current 34]
adb -s <serial> shell dumpsys package com.lightningpiggy.app | grep -E versionCode  # what's on device
APP_VARIANT=production eas build:version:get --platform android --profile production  # what's on server
----
If the server counter is < device versionCode, bump it (`set`) and rebuild. The APK in hand cannot be re-stamped — `versionCode` is signed into the manifest, repackaging breaks the EAS keystore signature.

| "How long is the EAS iOS build queue right now?" — no public API
| EAS doesn't expose live queue depth via API; status.expo.dev only reports operational/degraded. Best signals: +
1. **Last completed-build duration** — proxy for current load: +
[source,bash]
----
NODE_NO_WARNINGS=1 eas build:list --platform ios --status finished --limit 1 --non-interactive \
  | grep -E "Started at|Finished at"
----
Subtract the two timestamps. Free-tier iOS production has historically been ~2-3 hours wall-time. +
2. **Are any builds currently queued?** +
[source,bash]
----
NODE_NO_WARNINGS=1 eas build:list --status in-queue --status in-progress --limit 10 --non-interactive
----
Empty output = nothing in flight; positive output shows projects ahead of you in the shared queue. +
3. **EAS dashboard** — https://expo.dev/accounts/bengweeks/projects/lightning-piggy-app/builds renders queue position client-side once a build is in flight; the initial HTML doesn't include it. +
*Why this matters:* GH-Release-driven iOS builds go to TestFlight after EAS finishes + Apple processes (~10 min). For a same-day TestFlight push, kick off in the morning. For a v1.X.Y-rc1 cycle, plan ~3-4 hours from tag-push to "installable in TestFlight". +
*The only definitive number* is the queue position EAS prints once your build is submitted. Until then, all numbers are estimates from prior runs.

| Maestro can't tap a third-party signer's FAB / button (Aegis, Clave, etc) — `Element not found` or silent no-op
| Some signer apps (notably Aegis ≥0.5.0, possibly Clave) ship without accessibility labels on key UI elements like the "+" FAB to add a connection. Maestro's `tapOn: { id: ... }` and `tapOn: { text: ... }` both miss because there's nothing to match. The `index:` selector also ranks unpredictably for un-labelled `ImageView` clickables. +
*Why coordinates aren't a workaround:* `point: '90%, 91%'` and `adb shell input tap` are forbidden by the project's CLAUDE.md (fragile across screen sizes / OS versions / device profiles). +
*Workarounds:* +
1. Drive the signer setup manually before kicking off the Maestro flow — pre-pair the signer with the test nsec via its own UI, then have Maestro start from a state where the signer is ready. +
2. File the missing a11y label upstream and pin the older signer version where the bug doesn't apply. +
3. Use a signer that exposes a11y selectors (per the 2026-05 review, Nowser surfaces tappable elements with `accessibilityLabel` / resource-id, so Maestro can drive its connection-add flow without coordinate fallbacks). +
*Validated this session:* PR #284 (NIP-46 / Clave) reaches the QR pairing screen + verifies the spec-compliant `nostrconnect://` URI fully via Maestro; the bunker-side approval cannot be Maestro-driven on Aegis until Aegis ships a `testTag` / `accessibilityLabel` on the FAB. Manual handoff is the supported path today.

| Metro not connecting to device over USB
| Run `adb reverse tcp:8081 tcp:8081` for USB port forwarding before launching the app.

| React Native DevTools fails on Linux with `chrome-sandbox ... mode 4755`
| Pressing `j` in Metro launches React Native DevTools (a bundled Chromium). On Ubuntu 24.04+, unprivileged user namespaces are disabled, so the SUID sandbox helper must be root-owned with mode 4755. The binary ships inside `~/.cache/dotslash/...` with the wrong permissions. +
*Fix:* copy the **full `chrome-sandbox` path printed in the error** (do not construct it by hand) and run: +
1. `sudo chown root:root <full_path_to_chrome_sandbox>` +
2. `sudo chmod 4755 <full_path_to_chrome_sandbox>` +
Example path format: `~/.cache/dotslash/cb/<hash>/React Native DevTools-linux-x64/chrome-sandbox`. +
*Caveats:* this marks a cached binary as setuid-root, which is security-sensitive — apply only to the exact path from the error, never to other files. The hash segment changes when DevTools updates or the dotslash cache is cleared, so the fix must be re-applied against the new path. This does not cause a blank screen on the device — DevTools is a host-side debugger only.

| App shows black screen or "Unable to load script" after force-stop
| The Metro bundler connection can break when force-stopping the app. Fix: (1) Kill Metro with `pkill -f "expo start"`, (2) restart with `npm start` (or `npx expo start --clear` to also clear cache), (3) run `adb reverse tcp:8081 tcp:8081` again, (4) relaunch the app. If still black, do a full rebuild with `npx expo run:android`.

| Red-screen "Failed to compile / None of these files exist" pointing at a deleted module after a pull or merge
| Metro's transform cache keys on **file path**, not file hash. When you `git pull` (or merge) a branch that deletes or renames a module, the old transformer entries still resolve — so the next rebuild happily re-emits a phantom `require('./icons/ZapIcon')` even though the source has already been migrated (e.g. to `lucide-react-native`). The source view in the red-screen is the NEW file; the resolver is chasing the OLD graph. Restart Metro with `--clear` to wipe `$TMPDIR/metro-cache/*`: +
[source,bash]
----
pkill -9 -f "expo start"
npx expo start --dev-client --clear
----
Always follow merges that add/remove icon or asset files with a `--clear` restart. Seen on 2026-04-21 after merging PR #96 (lucide icon migration deleted `icons/{Zap,Copy,Share}Icon.tsx`).

| Claude Code warns "image exceeds the dimension limit for many-image requests (2000px)"
| Screenshots captured with `adb exec-out screencap -p` can exceed 2000px on high-DPI devices. Before sending to Claude, resize: `adb exec-out screencap -p > /tmp/screen.png && convert /tmp/screen.png -resize 1200x1200\> /tmp/screen_small.png`. Or run `/compact` to clear old images from context, or start a new conversation.

| Characters dropped / visually deleted while typing into a `TextInput` inside a `@gorhom/bottom-sheet` modal
| Plain React Native `TextInput` loses keystrokes when placed inside `BottomSheetView` / `BottomSheetModal` — the sheet container is animated by `react-native-reanimated` and can re-render while the IME is mid-composition, dropping characters. The symptom looks like the input "friendly deletes" letters you just typed. +
*Fix:* use `BottomSheetTextInput` from `@gorhom/bottom-sheet` (not `TextInput` from `react-native`) for every input inside a bottom-sheet subtree. It hooks into the sheet's keyboard + animation context and keeps keystrokes intact. All sheets in this repo (FeedbackSheet, TransferSheet, AddWalletWizard, EditProfileSheet, AddFriendSheet, NostrLoginSheet, and — as of #96 — SendSheet) follow this pattern. If you add a new sheet, grep `BottomSheetTextInput` for the canonical import site.

| Maestro `inputText` drops characters in plain `TextInput`
| Generic char-drop, not the BottomSheetTextInput-specific failure below. Maestro's `adb shell input text "$value"` synthesizes keystrokes faster than Android's IME composition pipeline can absorb them — long strings (>8-12 chars) lose chars in the middle. Affects the SDK 55 New Architecture especially. +
*Workaround:* split the input into ~8-character chunks across multiple `inputText` steps:
[source,yaml]
----
- inputText: "justowl3"
- inputText: "@primal"
- inputText: ".net"
----
The IME finishes composing each chunk before the next one starts. For nsec / npub / NWC URLs (60-90 chars) this is too painful — use the clipboard-paste pattern (next entry). +
*When chunking is fine:* short text (lightning addresses, group names, search queries). When clipboard-paste is required: any 30+ char secret / URL.

| Maestro `inputText` doesn't update React state in `BottomSheetTextInput`
| `BottomSheetTextInput` from `@gorhom/bottom-sheet` doesn't always trigger `onChangeText` when Maestro uses `inputText`. The text appears visually in the input but the React state stays empty, causing validation to fail. Under React Native's New Architecture (Fabric, default in SDK 55) the `testID` lands on the wrapper View rather than the native `EditText`, so the synthetic `adb shell input text` keystrokes don't advance the JS event path. **Do not** replace with regular `TextInput` — the bottom sheet needs its own input to adjust when the keyboard opens. +
*Canonical workaround — clipboard paste:* expose an in-app "Paste" button next to the input that calls `Clipboard.getStringAsync()` (expo-clipboard) and sets state directly. From the Maestro host, seed the clipboard first — host→emulator clipboard sync forwards it to the device (confirmed working on Android 16 emulator with Android Studio's default clipboard-sharing setting):

[source,bash]
----
source .env && echo -n "$MAESTRO_NWC1" | wl-copy   # Wayland
# or
echo -n "$MAESTRO_NWC1" | xclip -selection clipboard  # X11
maestro test tests/e2e/test-add-wallet-via-paste.yaml
----

The Maestro flow then taps the Paste button (`id: 'wizard-nwc-paste'`) instead of using `inputText`, and state updates cleanly through the normal React setter path. See `AddWalletWizard.tsx` for the canonical implementation and `tests/e2e/test-add-wallet-via-paste.yaml` for the flow. +

*Why `pasteText` (Maestro's built-in) doesn't help:* Maestro's `pasteText` / `copyTextFrom` use an **internal shim clipboard** (not Android's `ClipboardManager`) and still synthesize key events under the hood — so they hit the exact same wrapper-targeting failure. Seeding the real device clipboard and tapping an in-app Paste button bypasses both the shim and the keystroke path. Confirmed via the Maestro docs (https://docs.maestro.dev/api-reference/commands/pastetext) and feature-request https://github.com/mobile-dev-inc/Maestro/issues/2029. Deep-link seeding (`app://.../path?value=...`) is the only other approach that bypasses text entry entirely — viable if a Paste button isn't desired in the UI.

| Maestro carousel swipes trigger notification shade or leave the app
| Horizontal swipes at the top of the screen (y < 25%) can pull down the notification shade or swipe away the app. Use `y: 30%` or higher for carousel swipes. The wallet carousel sits at roughly 15-25% from the top, so swipe at `start: 85%, 30%` / `end: 15%, 30%` to target it safely.

| `tiny-secp256k1` fails to resolve in React Native (WASM not supported)
| Hermes doesn't support WebAssembly. Use `@bitcoinerlab/secp256k1` (pure JS) instead. This is the same approach used by BlueWallet. See `metro.config.js` for the custom resolver needed for `@noble/curves` ESM exports.

| Metro can't resolve `@noble/curves/secp256k1` or `@noble/hashes/sha2`
| These packages use ESM `exports` maps with `.js` extensions that Metro doesn't handle by default. Add a custom resolver in `metro.config.js` that appends `.js` to `@noble/*` subpath imports. Also add `'cjs'` to `resolver.sourceExts` for `bitcoinjs-lib`'s CommonJS entry point.

| `Buffer.equals is not a function` in React Native
| The `buffer` npm polyfill doesn't implement all Node.js Buffer methods. Use `Uint8Array` operations or hex string comparison instead of `Buffer.equals()`, `Buffer.copy()`, etc.

| `Invalid network version` when importing zpub or ypub
| `bip32.fromBase58()` only recognises `xpub` version bytes by default. Convert zpub/ypub to xpub by swapping the 4-byte version prefix (see `toXpub()` in `onchainService.ts`). The key data is identical across all three formats.

| Maestro `clearState` wipes all app data (wallets, Nostr login, settings)
| `clearState` deletes AsyncStorage and SecureStore. Only use it when you want a truly fresh install. For tests that need existing wallets, use `launchApp` without `clearState`.

| NWC "failed to publish: All promises were rejected"
| The LNbits Nostrclient relay proxy does not send NIP-20 OK responses after receiving EVENT messages (see https://github.com/lnbits/nostrclient/issues/52[lnbits/nostrclient#52]). The `@getalby/sdk` uses `nostr-tools` `SimplePool.publish()` which waits for an OK response from the relay — when none comes, it times out (4.4s) and rejects with `AggregateError`. +
*Fix:* `nwcService.ts` monkey-patches the relay pool's `publish` method to resolve immediately (fire-and-forget) since the NWC response comes back via the subscription, not the OK acknowledgement. This patch is applied during `connect()`.

| NWC "reply timeout" after publish succeeds
| The NWC request event was published to the relay but the wallet's response didn't arrive within the SDK's reply timeout (~10s). This can happen when the LNbits NWC extension is slow to process requests, or the relay is slow to forward the response event. +
*Workaround:* The `withRetry` helper retries with exponential backoff (2s, 4s, 8s). If persistent, check the LNbits NWC extension is running and the Nostrclient relay has active upstream relay connections (green status indicators in the Nostrclient admin).

| NWC relay intermittently drops WebSocket connections
| The `@getalby/sdk` opens a WebSocket to the NWC relay during `enable()`. If the connection drops (idle timeout, network change), all subsequent requests fail with "failed to publish". +
*Fix:* `nwcService.ts` detects publish failures and automatically reconnects by creating a fresh `NostrWebLNProvider` with the stored NWC URL. The reconnect logic is in `getBalance()`, `payInvoice()`, and `listTransactions()`.

| App shows "Unable to load script" on physical device
| The device can't reach Metro bundler. Run `adb reverse tcp:8081 tcp:8081` to forward the port from the device to the host machine. Then tap RELOAD (R, R) on the device.

| Black screen after reload / app launch (eventually loads)
| The NWC wallet connection retries block the loading spinner. Each wallet attempts `getBalance()` with 3 retries and exponential backoff (2s, 4s, 8s) = ~14s per wallet. With 2 NWC wallets, startup can take ~30 seconds before the UI renders. The wallet is marked "Connected" after `enable()` succeeds (not after `getBalance()`), but the loading state (`isLoading`) waits for all wallets to finish connecting. +
*Workaround:* Force restart the app and wait ~30-40 seconds. If persistent, check the LNbits NWC relay is running.

| LNbits "Payment in flight" log spam blocks NWC processing
| A Lightning payment stuck in `IN_FLIGHT` status causes LNbits to poll LND at ~16 checks/second with no backoff, producing 50,000+ log lines/hour and starving the asyncio event loop. NWC, Nostrclient, and all WebSocket-based extensions stop responding while the HTTP UI continues working. +
*Fix:* Apply the exponential backoff patch: +
1. `cat scripts/fix_lnbits_inflight.py \| ssh black-panther "cat > /tmp/fix.py"` +
2. `ssh black-panther "docker cp /tmp/fix.py lnbits-family:/tmp/ && docker exec lnbits-family python3 /tmp/fix.py"` +
3. `ssh black-panther "docker restart lnbits-family"` +
The patch modifies `get_payment_status()` in `lnbits/wallets/lndrest.py` to `continue` on IN_FLIGHT status instead of returning immediately, keeping the LND TrackPaymentV2 stream open until the payment resolves. Also adds a 300s stream timeout. Must be re-applied after LNbits upgrades. See https://github.com/lnbits/lnbits/issues/3917[lnbits/lnbits#3917], PR https://github.com/lnbits/lnbits/pull/3918[lnbits/lnbits#3918].

| "Cannot read property 'reload' of undefined" on BDK send
| This error is NOT from BDK — it's from React Native's `DevSettings` module. The actual error (Electrum connection failure) is swallowed, and RN's error handler tries to call `NativeDevSettings.reload()` which is null in Expo dev-client builds. +
*Fix:* `onchainService.ts` now forces a fresh Electrum connection (`blockchain = null`) before every `sendTransaction()` call. This avoids stale cached connections that cause the underlying Electrum error. The "reload" symptom disappears because the Electrum call succeeds. +
*Diagnosis:* Add `[BDK]` console.log statements before/after each step (sync, broadcast) to identify which step fails. The "reload" error masks the real cause.

| Boltz reverse swaps (LN → on-chain) occasionally fail with `pay_invoice` timeout
| Boltz Exchange dynamically selects which LND node issues reverse swap invoices based on amount. Observed on 2026-04-12:

* **≤ 250,000 sats**: invoice issued by "Boltz Mini" (pubkey `03e9c515b1e48c0e3bff9af8af1fce15d9b3f1ebb51e0311b1f80e820ba4cabb18`, 17 public channels, 4.18 BTC capacity)
* **≥ 250,001 sats**: invoice issued by the main Boltz LND node (pubkey `026165850492521f4ac8abd9bd8088123446d126f648ca35e60f88177dc149ceb2`)

Boltz Mini IS publicly routable, but initial swap attempts failed because LND's route pathfinding defaulted to `nicehash-ln1 → kappa → Boltz Mini` which had a stuck HTLC problem at kappa. After opening a direct channel to the main Boltz node, LND's graph eventually learned the `Boltz → Boltz Mini` path and routing works. +

*Mitigations:*

* Open a direct channel to the main Boltz node — gives LND a reliable path to Boltz Mini (Boltz → Boltz Mini)
* Penalise the problematic intermediate node in mission control: `lncli importmc --failure --force <problem_node> <dest> $(date +%s) <amt_msat>`
* Wait for gossip to propagate new channel info (can take minutes)
* For persistent failures at a specific amount, retry — Boltz node selection may rotate

*Boltz response:* Node selection is dynamic and not exposed in the API. Any assigned Boltz node should be routable. Failures indicate a local routing/topology issue, not a Boltz-side routing hint problem. +

*Status:* Working after direct channel to main Boltz established.

| Electrum connection drops or broadcast fails silently
| On-chain sync and broadcast use BDK (`bdk-rn`) now — the old `electrum-client` / `react-native-tcp-socket` stack is gone. Stale Electrum sockets can still manifest as confusing errors like "Cannot read property 'reload' of undefined" (a React Native DevSettings crash that masks the underlying BDK error). +
*Fix:* `onchainService.broadcastRawTx` and `sendTransaction` force `blockchain = null` before each operation to open a fresh connection; `disconnectElectrum()` is called when the user changes Electrum server in Account settings. If errors persist, force-restart the app.

| NIP-20 OK fix lost after LNbits upgrade or container recreate
| The Nostrclient extension bundled with the LNbits Docker image does not include our NIP-20 OK fix (PR https://github.com/lnbits/nostrclient/pull/68[lnbits/nostrclient#68]). Without it, NWC clients using `@getalby/sdk` timeout waiting for OK responses after publishing events. +
*To re-apply the fix after upgrading LNbits:* +
1. Copy the fix script into the container: `cat apply_nip20_fix.py \| ssh black-panther "cat > /tmp/apply_nip20_fix.py"` +
2. Run inside container: `ssh black-panther "docker cp /tmp/apply_nip20_fix.py lnbits-family:/tmp/ && docker exec lnbits-family python3 /tmp/apply_nip20_fix.py"` +
3. Restart: `ssh black-panther "docker restart lnbits-family"` +
4. Verify: `ssh black-panther "docker exec lnbits-family grep 'NIP-01: OK' /app/lnbits/extensions/nostrclient/router.py"` +
The fix script (`apply_nip20_fix.py`) patches `router.py` to send `["OK", event_id, true, ""]` after publishing EVENT messages, and `["OK", event_id, false, "error: no relay connections"]` when no relays are connected.

| `Security returned a non-successful error code: 36` during `eas build` or `eas submit` over SSH
| The Mac's login keychain is locked because there's no active GUI session — `fastlane` can't save the Apple ID password so it treats auth as failed, even when the password is correct. +

The settings override (`Settings.Secure.show_ime_with_hard_keyboard=1`) is supposed to force Gboard despite that, but in practice this is intermittent on the Pixel-8 / Android 16 system image — Gboard reads the setting lazily and the IMM caches its IME-binding state across IME process restarts, so toggling the setting while the system is running often does nothing. +
*Initial fix (works briefly):* edit the AVD config to drop the explicit hw-keyboard flag, then **cold-boot** the emulator and ensure the override is set BEFORE the first input focus:
[source,bash]
----
# `sed -i` is portable across GNU (Linux) and BSD (macOS) sed only if
# you branch on `uname` — BSD sed requires an explicit (possibly empty)
# backup-suffix argument after `-i`, GNU sed forbids it.
if [ "$(uname)" = "Darwin" ]; then
  sed -i '' 's/^hw\.keyboard=yes$/hw.keyboard=no/' ~/.android/avd/<AVD_NAME>.avd/config.ini
else
  sed -i 's/^hw\.keyboard=yes$/hw.keyboard=no/' ~/.android/avd/<AVD_NAME>.avd/config.ini
fi
adb -s emulator-5554 emu kill
$ANDROID_HOME/emulator/emulator -avd <AVD_NAME> -no-snapshot-load &
adb wait-for-device
adb shell settings put secure show_ime_with_hard_keyboard 1
----
After cold boot, tapping a text input shows the full docked Gboard at the bottom and `react-native-keyboard-controller` reports the correct IME height. +
*Important: this is NOT a permanent fix.* `adb shell dumpsys input` after cold boot still shows `Device 2: AT Translated Set 2 keyboard` regardless of the AVD setting — QEMU always attaches a virtual PS/2 keyboard. Over the life of an emulator session the floating IME chooser will return after some combination of: re-launching the app, switching IMEs, running Maestro tests that programmatically inject input, or simply re-focusing an input after backgrounding. When this happens, you have to cold-boot again — `adb reboot`, force-stopping LatinIME, re-toggling `show_ime_with_hard_keyboard`, and `am force-stop` of the system UI all fail to recover the docked keyboard once the IMM has cached the suppressed state. +
*Side effect:* with `hw.keyboard=no`, the laptop keyboard no longer types into the emulator. For interactive development this is annoying — your two paths are: (a) tap-input via mouse on the emulator window and use Gboard, or (b) inject text from the host CLI with `adb -s emulator-5554 shell input text "your%smessage"` (`%s` = space). For Maestro tests, prefer the `inputText` action which doesn't depend on a visible IME. +
*Workaround when neither approach works:* if you need both laptop typing AND a working IME, run on a real device. For Maestro tests run inside CI, use `inputText` exclusively and never assert on the keyboard's visible bounds — only on the visibility / position of the composer above where the IME *would* be. +
*Diagnostic:* after cold boot `adb shell dumpsys input_method | grep mInputShown` should be `true` while focused, and `adb shell dumpsys input | grep -E "AT Translated|qwerty"` will always show the AT keyboard regardless. The reliable test is whether tapping an input renders the docked Gboard pixels — there is no clean adb signal for "Gboard is rendered vs the floating chooser is rendered".

| Emulator boots to a grey screen / `boot_completed` never flips to 1 / `adb devices` flaps `device` ↔ `offline` indefinitely
| The persistent userdata overlay (`userdata-qemu.img.qcow2`) has accumulated corrupt state from a prior crashed boot — typically an Android Studio's *Digital Wellbeing* ANR cascade, a hard-killed `qemu-system` process, or a Vulkan/gfxstream stall during the initial GPU init. Once that file is corrupt every subsequent cold boot tries to mount it and either hangs in `system_server` init, never reaches `getprop sys.boot_completed=1`, or — most commonly — gets far enough that ADB sees the device but the framework never stabilises, so `adb` reports `device` for a few seconds then drops back to `offline` as the watchdog fires. Switching GPU modes (`-gpu host` → `-gpu swiftshader_indirect`) does **not** help — the corrupt overlay is the root cause, not the renderer. Symptom in the emulator log: GPU init completes (`Setting display: 0 configuration to: 1080x2400`, `Created extended window`) but every subsequent `adb command '… getprop sys.boot_completed' failed: 'adb: device offline'` line repeats forever. +
*Fix (factory-reset just the runtime state, no need to recreate the AVD):* kill any stuck instances, delete the qcow2 overlays + snapshots + lock files, then cold-boot. The base `.img` files regenerate fresh overlays on first boot — equivalent to `avdmanager` recreating the AVD without actually needing `cmdline-tools` installed. +
[source,bash]
----
# 1. Kill any stuck emulator processes
adb emu kill 2>/dev/null
pkill -9 -f qemu-system
pkill -9 -f "emulator -avd"

# 2. Back up config.ini (cheap insurance against the sed-on-shutdown
#    truncation bug noted in the "System UI isn't responding" entry)
cp ~/.android/avd/<AVD_NAME>.avd/config.ini \
   ~/.android/avd/<AVD_NAME>.avd/config.ini.preboot-bak

# 3. Delete the qcow2 overlays, snapshots, and runtime locks.
#    The base *.img files are NOT touched — they regenerate the
#    overlays on next boot.
rm ~/.android/avd/<AVD_NAME>.avd/userdata-qemu.img.qcow2
rm ~/.android/avd/<AVD_NAME>.avd/cache.img.qcow2
rm ~/.android/avd/<AVD_NAME>.avd/sdcard.img.qcow2
rm ~/.android/avd/<AVD_NAME>.avd/encryptionkey.img.qcow2
rm -f ~/.android/avd/<AVD_NAME>.avd/multiinstance.lock
rm -f ~/.android/avd/<AVD_NAME>.avd/read-snapshot.txt
rm -f ~/.android/avd/<AVD_NAME>.avd/quickbootChoice.ini
rm -rf ~/.android/avd/<AVD_NAME>.avd/snapshots
rm -rf ~/.android/avd/<AVD_NAME>.avd/tmpAdbCmds

# 4. Cold-boot. Hardware GPU works fine once the overlay is clean.
QT_QPA_PLATFORM=xcb $ANDROID_HOME/emulator/emulator \
  -avd <AVD_NAME> -gpu host -no-snapshot -no-boot-anim
----
*What you lose:* every app installed inside the AVD, signed-in accounts, AVD-side preferences. The AVD config (`config.ini`, `hardware-qemu.ini`, `AVD.conf`) and base system image are preserved, so the AVD itself stays valid. +
*What you gain:* the `userdata-qemu.img.qcow2` file goes from a multi-GB accumulated overlay (often 7 GB+ after months of use) back to a thin overlay regenerated from the system image's userdata. +
*Don't waste time on these false leads first:* (a) `adb reboot` won't help — the corrupt overlay is what re-mounts on every boot. (b) Switching GPU mode won't help — the GPU init phase succeeds; the failure is in Android userspace boot. (c) Increasing AVD RAM won't help — `system_server` isn't OOM, it's hanging on the corrupt overlay. (d) Re-running `expo prebuild` won't help — this is an emulator state issue, not an app build issue. +
*Verify clean boot:* `adb -s emulator-5554 shell getprop sys.boot_completed` returns `1` (with a stable trailing newline) and `adb devices` shows `device` (not `offline`) consistently for >30 s. UI rendering should be smooth — if you see frame stutters and a `Process system isn't responding` dialog, you're probably on `swiftshader_indirect` (software GL) and need to re-launch with `-gpu host` now that the overlay is clean.

| Maestro triple-tap (e.g. version label to enable dev mode) silently fails on API 36 emulator
| Maestro's official supported ceiling is API 34 — three sequential `tapOn` calls on an API 36 image take >1 s end-to-end, exceeding the 1-second window in `handleVersionTap` (and similar in-app tap-counter handlers like add-wallet hot wallet unlock). Symptom: 3 `tapOn`s complete cleanly per Maestro logs but the dev-mode alert never appears. The same flow can also be defeated by an API 36 PackageManager bug that breaks RN dev builds (`Activity class does not exist` on `am start`) — independent of installer attribution, install path, or `adb reboot`. +
*Fix:* Run E2E + dev work on **API 34**, **`google_apis`** image (NOT `google_play` — saves ~300 MB RAM + the GMS daemons that thrash `system_server`), **`Medium Phone`** device profile (Pixel 9a / Pixel 10 device profiles have a known input-swallowing bug on Linux/macOS/Windows). Bump RAM + heap before first boot so `system_server` doesn't crash under a 150 MB+ RN dev APK install: +
[source,bash]
----
# Install the image once
$ANDROID_HOME/cmdline-tools/latest/bin/sdkmanager \
  "system-images;android-34;google_apis;x86_64"

# Create the AVD
echo no | $ANDROID_HOME/cmdline-tools/latest/bin/avdmanager create avd \
  -n LP_API34 -k "system-images;android-34;google_apis;x86_64" \
  -d "medium_phone" --force

# Bump RAM + heap (default 1.5 GB / 228 MB OOMs on RN dev installs;
# 4 GB / 1024 MB matches satmo-react-native's validated config)
sed -i 's/^hw.ramSize=.*/hw.ramSize=4096/' \
  ~/.android/avd/LP_API34.avd/config.ini
sed -i 's/^vm.heapSize=.*/vm.heapSize=1024/' \
  ~/.android/avd/LP_API34.avd/config.ini

# Cold-boot
$ANDROID_HOME/emulator/emulator -avd LP_API34 \
  -no-snapshot-load -no-snapshot-save -no-audio -gpu host
----
*Disk-space follow-up:* if `adb install` later fails with `INSTALL_FAILED_INSUFFICIENT_STORAGE` (the dev APK + Hermes intermediates need 3-4 GB free), edit `config.ini` to set `disk.dataPartition.size=12G` (default 800M-6G), then cold-boot with `-wipe-data -no-snapshot` (destructive — wipes installed apps + accounts). +
*Performance bonus:* API 34 lacks the stylus-handwriting service that API 35+ ship, removing one of the two soft-keyboard suppression gates documented above. Re-running `adb shell settings put secure stylus_handwriting_enabled 0` on every cold boot becomes unnecessary. +
*Reference:* `../satmo-react-native/docs/TROUBLESHOOTING.adoc` — "Emulator with API 36 (Android 16) image refuses to launch the dev build" and "Android emulator System UI isn't responding".

| AVD `system_server` crashes / "System UI isn't responding" *after* installing a fresh dev APK
| Default AVD allocation (RAM 1.5 GB, Dalvik heap 228 MB) is undersized for installing a 150 MB+ React Native dev APK. `kswapd0` burns 40% kernel CPU, `system_server` dies, Binder IPC fails with `No space left on device` (RAM-pressure symptom, NOT disk), every subsequent `adb install` / `pm list packages` / app launch fails until guest reboot. *Prevent recurrence:* use a `google_apis` system image (NOT `google_play`) — drops the Play Store + GMS daemons that consume ~300 MB RAM / 60% CPU. The Play *Services* subset (Google Sign-In, Firebase Auth, FCM) still works; only the storefront and its background daemons are excluded. +
*Reference:* `../satmo-react-native/docs/TROUBLESHOOTING.adoc` — "Android emulator System UI isn't responding / app stuck on splash" and "AVD `system_server` crashes during `adb install`".

| Claude Code Bash tool fails silently (exit 1 or 134, empty stdout/stderr) when `/tmp` is full
| Claude Code creates a shell snapshot in `/tmp` at session start; when the volume is full, every Bash tool invocation silently fails. `bash --noprofile --norc -c '…'` and the file-redirect workaround `{ cmd; } > /tmp/out 2>&1; exit 0` also produce no output (the spawned bash never reaches the redirect). `/doctor` does not flag it; only the Write tool surfaces the underlying `ENOSPC` honestly. On a typical Linux dev machine `/tmp` is a tmpfs sized at 50 % of RAM; common culprits are accumulated git worktrees in `/tmp/wt-*` (1-2 GB of `node_modules` each, up to 7+ GB with EAS local-build artifacts), `/tmp/metro-cache`, `/tmp/node-compile-cache`, and signed-but-not-cleaned `*.apk` outputs. +
*Diagnose:* `df -h /tmp` to confirm full; `du -shx /tmp/* 2>/dev/null \| sort -h \| tail -20` to identify culprits. +
*Fix:* `rm -rf /tmp/wt-* /tmp/metro-* /tmp/node-compile-cache /tmp/*.apk` (review the list first; these are caches and ephemeral build outputs). Then `git worktree prune` from any repo whose worktrees lived in `/tmp` so the admin entries match reality. Restart Claude Code so it can write a fresh shell snapshot. +
*Prevent recurrence:* don't create git worktrees in `/tmp` on tmpfs machines. Use `git worktree add ~/.local/share/wt-<name> <branch>` (or `~/wt-*`) — survives reboot and doesn't compete with the tmpfs cap. +
*False leads to skip:* reinstalling Claude Code, killing leftover `chrome-native-host` processes, removing dangling plugin entries from `~/.claude/settings.json`, switching out of Remote Control mode — none of these fix the underlying full-tmpfs.

| Android 16 KB page-size compatibility warning on Pixel 8 / Android 16
| Local debug builds via `npm run android:dev` show an Android-system "this app isn't 16 KB-compatible" dialog before any LP UI renders.
