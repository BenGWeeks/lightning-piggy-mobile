= Troubleshooting

[cols="1,2", options="header"]
|===
| Problem | Solution

| Bottom sheet doesn't slide up when keyboard opens on Android
| All of these are required: (1) Use `BottomSheetTextInput` from `@gorhom/bottom-sheet` instead of React Native's `TextInput`. (2) Use `BottomSheetScrollView` (not `BottomSheetView`) as the content wrapper. (3) Set `keyboardBehavior="interactive"`, `keyboardBlurBehavior="restore"`, and `android_keyboardInputMode="adjustResize"` on the `BottomSheetModal`. (4) Ensure `android:windowSoftInputMode="adjustResize"` in AndroidManifest (via `plugins/withAdjustResize.js`). (5) Add a `Keyboard.addListener` to track keyboard height and set dynamic `paddingBottom` on the `contentContainerStyle` (e.g. `paddingBottom: keyboardHeight > 0 ? keyboardHeight + 80 : 40`). (6) Do NOT wrap content in `TouchableWithoutFeedback` with `Keyboard.dismiss` — this interferes with the sheet's keyboard handling. See `NostrLoginSheet.tsx` as the reference implementation.

| Maestro can't find elements behind a modal Alert dialog
| Dismiss the alert first (e.g. tap "KEEP CURRENT" or "Cancel") before asserting visibility of elements underneath.

| Maestro `inputText` produces invalid bech32 checksum on secure fields
| Add `.trim()` to the input value before processing. Maestro may append whitespace characters to secure text fields.

| Profiles not loading for some Nostr contacts
| Add more relay aggregators to `PROFILE_RELAYS` in `nostrService.ts` (e.g. `relay.primal.net`, `purplepag.es`). Use smaller batch sizes (50) with longer timeouts (12s) for `fetchProfiles`.

| App unresponsive after Nostr login (20s lockup)
| Wrap `setContacts` calls in `startTransition()`, use `InteractionManager.runAfterInteractions` to defer network I/O, add `freezeOnBlur: true` to tab navigator, and yield to event loop between profile fetch batches with `setTimeout(r, 50)`.

| `npx expo start` fails but `npm start` works
| The `start` script includes `--dev-client` which is required for custom native modules. Always use `npm start`.

| Metro not connecting to device over USB
| Run `adb reverse tcp:8081 tcp:8081` for USB port forwarding before launching the app.

| App shows black screen or "Unable to load script" after force-stop
| The Metro bundler connection can break when force-stopping the app. Fix: (1) Kill Metro with `pkill -f "expo start"`, (2) restart with `npm start` (or `npx expo start --clear` to also clear cache), (3) run `adb reverse tcp:8081 tcp:8081` again, (4) relaunch the app. If still black, do a full rebuild with `npx expo run:android`.

| Claude Code warns "image exceeds the dimension limit for many-image requests (2000px)"
| Screenshots captured with `adb exec-out screencap -p` can exceed 2000px on high-DPI devices. Before sending to Claude, resize: `adb exec-out screencap -p > /tmp/screen.png && convert /tmp/screen.png -resize 1200x1200\> /tmp/screen_small.png`. Or run `/compact` to clear old images from context, or start a new conversation.

| Maestro `inputText` doesn't update React state in `BottomSheetTextInput`
| `BottomSheetTextInput` from `@gorhom/bottom-sheet` doesn't always trigger `onChangeText` when Maestro uses `inputText`. The text appears visually in the input but the React state stays empty, causing validation to fail. **Do not** replace with regular `TextInput` — the bottom sheet needs its own input to adjust when the keyboard opens. Workarounds: (1) Use Maestro's clipboard paste approach, (2) tap the input field first to ensure focus before typing, (3) accept intermittent flakiness — the first test run in a session usually works.

| Maestro carousel swipes trigger notification shade or leave the app
| Horizontal swipes at the top of the screen (y < 25%) can pull down the notification shade or swipe away the app. Use `y: 30%` or higher for carousel swipes. The wallet carousel sits at roughly 15-25% from the top, so swipe at `start: 85%, 30%` / `end: 15%, 30%` to target it safely.

| `tiny-secp256k1` fails to resolve in React Native (WASM not supported)
| Hermes doesn't support WebAssembly. Use `@bitcoinerlab/secp256k1` (pure JS) instead. This is the same approach used by BlueWallet. See `metro.config.js` for the custom resolver needed for `@noble/curves` ESM exports.

| Metro can't resolve `@noble/curves/secp256k1` or `@noble/hashes/sha2`
| These packages use ESM `exports` maps with `.js` extensions that Metro doesn't handle by default. Add a custom resolver in `metro.config.js` that appends `.js` to `@noble/*` subpath imports. Also add `'cjs'` to `resolver.sourceExts` for `bitcoinjs-lib`'s CommonJS entry point.

| `Buffer.equals is not a function` in React Native
| The `buffer` npm polyfill doesn't implement all Node.js Buffer methods. Use `Uint8Array` operations or hex string comparison instead of `Buffer.equals()`, `Buffer.copy()`, etc.

| `Invalid network version` when importing zpub or ypub
| `bip32.fromBase58()` only recognises `xpub` version bytes by default. Convert zpub/ypub to xpub by swapping the 4-byte version prefix (see `toXpub()` in `onchainService.ts`). The key data is identical across all three formats.

| Maestro `clearState` wipes all app data (wallets, Nostr login, settings)
| `clearState` deletes AsyncStorage and SecureStore. Only use it when you want a truly fresh install. For tests that need existing wallets, use `launchApp` without `clearState`.

| NWC "failed to publish: All promises were rejected"
| The LNbits Nostrclient relay proxy does not send NIP-20 OK responses after receiving EVENT messages (see https://github.com/lnbits/nostrclient/issues/52[lnbits/nostrclient#52]). The `@getalby/sdk` uses `nostr-tools` `SimplePool.publish()` which waits for an OK response from the relay — when none comes, it times out (4.4s) and rejects with `AggregateError`. +
*Fix:* `nwcService.ts` monkey-patches the relay pool's `publish` method to resolve immediately (fire-and-forget) since the NWC response comes back via the subscription, not the OK acknowledgement. This patch is applied during `connect()`.

| NWC "reply timeout" after publish succeeds
| The NWC request event was published to the relay but the wallet's response didn't arrive within the SDK's reply timeout (~10s). This can happen when the LNbits NWC extension is slow to process requests, or the relay is slow to forward the response event. +
*Workaround:* The `withRetry` helper retries with exponential backoff (2s, 4s, 8s). If persistent, check the LNbits NWC extension is running and the Nostrclient relay has active upstream relay connections (green status indicators in the Nostrclient admin).

| NWC relay intermittently drops WebSocket connections
| The `@getalby/sdk` opens a WebSocket to the NWC relay during `enable()`. If the connection drops (idle timeout, network change), all subsequent requests fail with "failed to publish". +
*Fix:* `nwcService.ts` detects publish failures and automatically reconnects by creating a fresh `NostrWebLNProvider` with the stored NWC URL. The reconnect logic is in `getBalance()`, `payInvoice()`, and `listTransactions()`.
|===
