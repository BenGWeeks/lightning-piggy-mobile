== Testing

This guide covers code quality checks, unit testing, and end-to-end testing for Lightning Piggy.

=== Code Quality

The project enforces code quality through TypeScript, ESLint, and Prettier. All checks must pass before committing.

[source,bash]
----
# TypeScript type checking
npx tsc --noEmit

# ESLint
npm run lint

# Prettier format check
npm run format:check

# Auto-format with Prettier
npm run format
----

=== Unit Testing with Jest

Unit tests run in Node (jsdom) via https://jestjs.io[Jest] with the `jest-expo` preset and https://callstack.github.io/react-native-testing-library/[`@testing-library/react-native`].

**Test files MUST live next to the source as `<file>.test.ts` / `<file>.test.tsx`** (co-located convention). The centralised `tests/unit/` directory is NOT scanned — `jest.config.js`'s `testMatch` is `<rootDir>/src/**/*.test.{ts,tsx}` only. Co-location surfaces missing tests at a glance, keeps tests with their subject when files move, and shortens the relative imports.

For cross-cutting tests (touching multiple files), put them next to the file whose API the test most directly exercises. Example: `src/services/nostrService.test.ts` covers the outgoing wire format; `src/utils/nip17Unwrap.test.ts` covers the incoming parser; `src/utils/syntheticGroupId.test.ts` covers the room-key derivation — even though all three relate to NIP-17 interop.

**Coverage scope:** `src/services`, `src/utils`, `src/contexts` only. Components are excluded (they're tested via Maestro pixel/flow tests instead — mocking Reanimated + bottom-sheet + `expo-image` for unit tests is high-effort, low-payoff).

==== Running Locally

[source,bash]
----
# Run the suite
npm test

# Run with coverage (writes ./coverage/coverage-summary.json plus a
# text summary to stdout — reporters in jest.config.js are
# `json-summary` + `text`; lcov is intentionally off to keep the
# artifact small)
npm run test:coverage

# Watch a single file
npx jest src/utils/foo.test.ts --watch
----

==== Coverage Gate (CI)

The `Coverage` workflow (`.github/workflows/coverage.yml`) runs on every PR to `main`:

. Runs `jest --coverage` on the PR branch.
. Re-runs `jest --coverage` on `main` for the baseline.
. Fails the PR if PR line-coverage drops below main by more than **0.5 percentage points** (the noise floor — small refactors that move lines around shouldn't trip the gate).

The first PR after this infrastructure merges may show a 0% main baseline if `coverage-summary.json` isn't generated yet — that's expected and the gate will pass trivially. Subsequent PRs are then held to the new baseline, so coverage can only steadily increase.

To tighten or loosen the gate, edit `NOISE_FLOOR` in the workflow file.

==== Picking What to Cover Next

Use `scripts/coverage-priorities.sh` to get a prioritized list of files most worth testing:

[source,bash]
----
# Generate coverage data first (otherwise every file ranks as 0% covered)
npm run test:coverage

# Top 20 ranked files (default)
./scripts/coverage-priorities.sh

# Top 50
./scripts/coverage-priorities.sh 50
----

The ranking heuristic is `(churn × loc × fanout) / (coverage% + 1)`:

* **churn** — distinct commits in the last 90 days (changes often → likely to regress)
* **loc** — file size (more behaviour → more to break)
* **fanout** — how many other src files reference this one (blast radius)
* **coverage** — already-tested files drop down the list

Pick from the top of the list. The script prints rank, score, and the underlying counts so you can see *why* a file ranked where it did.

=== E2E Testing with Maestro

End-to-end tests use https://maestro.mobile.dev[Maestro], a mobile UI testing framework. Tests are written in YAML and interact with the app through accessibility labels and test IDs.

==== Setup

[source,bash]
----
# Install Maestro
curl -Ls "https://get.maestro.mobile.dev" | bash

# Verify installation
maestro --version
----

Ensure the app is built and running on a connected device or emulator before running tests.

==== Running Tests

[source,bash]
----
# Run a single flow
maestro test .maestro/authentication/flow-006-login.yaml

# Run a whole feature area (numbered flows only)
maestro test .maestro/payments/

# Run the curated wallet/transfer suite (skips the un-numbered helpers
# under common/ and the perf flows, which aren't standalone-runnable)
source .env && bash .maestro/reporting/run-all.sh
----

NOTE: Don't run `maestro test .maestro/` over the whole tree — that
sweeps in the shared building blocks under `.maestro/common/` (and the
`perf/` flows), which are designed to be pulled in via `runFlow:` and
fail when executed on their own. Target a feature-area folder, a single
flow, or the `run-all.sh` runner instead.

==== Writing Tests

IMPORTANT: Never use coordinates (`point:`, `tapOn: { point: }`, `adb shell input tap`) in Maestro tests. Coordinates are fragile and break across screen sizes, devices, and OS versions.

Instead, always use accessibility-based selectors:

[cols="1,2,2", options="header"]
|===
| Selector | Usage | Example

| `id:`
| Match by `testID` prop
| `id: 'tab-friends'`

| `text:`
| Match by visible text or `accessibilityLabel`
| `text: 'Friends tab'`

|===

If a component is missing an accessibility label, add one to the source code rather than working around it with coordinates.

===== Composable Subflows

Subflows are reusable Maestro flows that live un-numbered in `.maestro/common/` (e.g. `common/login-as.yaml`). They aren't standalone tests — callers `runFlow:` them (via a relative path, e.g. `../common/login-as.yaml`) as a step inside a regular flow, so they're skipped by a feature-area run and must never be run on their own. Use them whenever multiple flows share a setup walk.

Available subflows (all under `.maestro/common/`):

[cols="2,3", options="header"]
|===
| Subflow | Purpose

| `common/login-as.yaml`
| Force-stop the app, sign out any prior session, then sign in via the nsec passed in `MAESTRO_NSEC_BIG`. Used by every multi-actor flow.

| `common/seed-test-wallet.yaml`
| Walk the AddWalletWizard to seed a test wallet so wallet-conditional tests (Send, Receive, Transfer, cached-tx, etc.) don't have to re-implement the wizard inline. Picks the wallet type based on which env var is non-empty:

`MAESTRO_TEST_NWC` → Lightning (NWC) wallet via connection string. +
`MAESTRO_TEST_MNEMONIC` → On-chain hot wallet via 12/24-word seed phrase. *Requires dev mode enabled in the app* — run `authentication/flow-004-login-dev.yaml` once first.

Required `WALLET_NAME` sets the alias shown on the wallet card. The flow asserts that alias appears in the carousel before returning. Maestro's `${VAR}` interpolation has no robust default-fallback across versions, so an unset value leaves the literal `${WALLET_NAME}` in the alias field and the visibility assertion never matches — pass it explicitly via `runFlow.env`.

|===

Example invocation from a wallet-conditional test:

[source,yaml]
----
- runFlow: ../common/login-as.yaml
- runFlow:
    file: ../common/seed-test-wallet.yaml
    env:
      MAESTRO_TEST_NWC: ${MAESTRO_NWC1}
      WALLET_NAME: "Seeded NWC"

# ... continue with wallet-dependent assertions ...
----

The required env vars (`MAESTRO_TEST_NWC`, `MAESTRO_TEST_MNEMONIC`) live alongside the rest of the Maestro fixtures in `.env.example`. Leave them blank in your local `.env` if you don't run wallet-conditional flows.

===== Accessibility Conventions

[cols="1,2", options="header"]
|===
| Component | Pattern

| Tab bar buttons
| `tabBarButtonTestID` (e.g. `tab-friends`) and `tabBarAccessibilityLabel` (e.g. `Friends tab`)

| Alphabet sidebar letters
| `testID` pattern `alphabet-{letter}` (e.g. `alphabet-M`)

| Interactive elements
| All buttons, tabs, and tappable items must have `accessibilityLabel` and/or `testID`

|===

=== Screenshots

Capture screenshots from a connected Android device using ADB:

[source,bash]
----
# Capture screenshot
adb exec-out screencap -p > /tmp/screen.png

# Resize for high-DPI devices (> 2000px)
convert /tmp/screen.png -resize 1200x1200\> /tmp/screen_small.png
----

The `\>` flag ensures ImageMagick only shrinks images that exceed 1200px — smaller images are left unchanged.
